# 项目需求：StateKanban — 基于状态看板的指令级开发引擎

## 项目概述

构建一个纯内存的 LLM 调度内核。LLM 降级为无状态算力单元，状态看板是唯一真相源，视口切片控制上下文，输出阀门保证写入安全。底座完全隔绝与底层操作系统的交互，内部进程通过消息总线高频对话，所有外部动作通过工具调用完成。

## 核心设计原则

1. **底座零 I/O**：内核不直接读写文件、不调 shell、不走网络。所有外部动作都是工具调用，结果通过返回值进入看板。
2. **看板即真相**：所有状态（意图、否决、确认、产出）只存在于看板数据结构中。进程崩溃后从看板恢复，不依赖文件系统。
3. **LLM 无状态**：每次调用 LLM 前由视口切片器裁剪极小上下文，LLM 不持有历史、不维护会话。
4. **流态/晶态分区**：未确认的意图和否决信号在流态区碰撞收敛；确认后的产出进入晶态区，不可篡改。
5. **输出阀门**：晶态区产出必须经过阀门校验（语法检查、类型检查、测试通过）才能物理写入。

## 架构分层

```
┌─────────────────────────────────────────┐
│           状态看板（唯一真相）            │
│  流态区：意图/否决信号                   │
│  晶态区：已确认的代码/配置               │
├─────────────────────────────────────────┤
│         消息总线（内部进程通信）          │
│  Coder ↔ Reviewer ↔ Integrator ...      │
├─────────────────────────────────────────┤
│         工具注册表 + 调度器              │
│  write_file / call_llm / run_shell /    │
│  read_file / search_code / ...          │
├─────────────────────────────────────────┤
│         视口切片器                       │
│  每次调用 LLM 前，裁剪极小固定阈值上下文  │
└─────────────────────────────────────────┘
         │
         ▼ （唯一出口）
    物理世界：硬盘 / API / Shell
```

## 核心模块

### 1. StateKanban（状态看板）

- 流态区（FluidZone）：意图信号、否决信号、碰撞记录。信号碰撞触发收敛循环。
- 晶态区（CrystalZone）：已确认的代码、配置、文档。只追加，不可修改。
- 视口索引：每个进程只能看到看板的一个切片，由视口切片器决定。
- 快照与恢复：支持看板序列化（JSON），进程崩溃后从快照恢复。

### 2. MessageBus（消息总线）

- 内部进程间的高频通信通道。
- 发布/订阅模式：进程订阅自己关心的信号类型。
- 同步调用 + 异步通知两种模式。
- 消息不落盘，纯内存。看板是持久化的唯一通道。

### 3. ViewportSlicer（视口切片器）

- 每次调用 LLM 前，根据进程角色和当前看板状态，裁剪出极小上下文。
- 固定阈值：单次 LLM 输入不超过 N tokens（可配置，默认 2000）。
- 切片策略：角色相关 > 依赖链上游 > 全局摘要。无关信息一律不传入。
- 切片日志：记录每次裁剪的输入/输出 token 数，用于优化切片策略。

### 4. OutputValve（输出阀门）

- 晶态区产出 → 阀门校验 → 物理写入。
- 校验链：语法检查 → 类型检查 → 测试执行 → 人工确认（可选）。
- 校验失败 → 信号回注流态区，触发返工。
- 原子写入：先写临时文件，校验通过后 rename。

### 5. ToolRegistry（工具注册表 + 调度器）

- 工具注册：write_file, read_file, run_shell, call_llm, search_code 等。
- 权限控制：每个进程只能调用声明的工具子集。
- 调用审计：所有工具调用记录到看板的审计区。
- 超时与重试：工具调用超时后信号回注流态区。

### 6. ProcessManager（进程管理）

- 进程生命周期：创建 → 激活 → 挂起 → 终止。
- 进程不能自杀，只能由调度器终止或被继任者替换。
- 心跳：进程定期向看板写入心跳，超时触发重启。
- 交接：新进程 claim_primary 后清理旧进程。

## 内部进程角色（初始集）

| 角色 | 职责 | 订阅信号 | 可调工具 |
|------|------|----------|----------|
| Coder | 代码生成 | intent.code, veto.code | write_file, read_file, search_code, call_llm |
| Reviewer | 代码审核 | crystal.code | read_file, search_code, call_llm |
| Integrator | 模块集成 | crystal.code, crystal.test | write_file, read_file, call_llm |
| Tester | 测试生成与执行 | crystal.code | write_file, run_shell, call_llm |
| Architect | 架构设计 | intent.arch | write_file, read_file, call_llm |

角色可扩展，通过工具注册表声明权限。

## LLM 调用层

### 初始形态：Claude Code CLI 适配

- 内核通过 subprocess 调用 `claude -p`，将视口切片后的上下文作为 prompt 传入。
- 返回值解析后注入看板。
- 这是过渡方案，内核成熟后废弃。

### 目标形态：Anthropic API 直调

- 内核直接调 Anthropic Messages API。
- 视口切片后的上下文作为 messages 数组传入。
- 支持 tool_use：LLM 返回工具调用意图，内核执行工具后将结果回注。
- 流式响应：支持 streaming，实时更新看板状态。

## 测试标准（论文定义）

1. **收敛率**：碰撞循环在 N 轮内收敛的比例。目标：90% 在 3 轮内收敛。
2. **拦截率**：恶意/错误代码被 Reviewer 拦截的比例。目标：95%+。
3. **无损交接**：进程崩溃后从看板恢复，状态零丢失。目标：100%。

## 技术栈

- 语言：Python 3.11+
- 异步：asyncio（消息总线 + 工具调度）
- LLM：anthropic SDK（API 直调）
- 序列化：JSON（看板快照）
- 测试：pytest + Mock LLM

## 项目边界

本系统不做：
- 图形界面
- 多用户权限
- 分布式部署（单机单进程）
- IDE 集成
- 代码仓库管理（git 操作由 run_shell 工具代理）
