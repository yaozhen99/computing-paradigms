# 项目需求：声纹映射输入法 (VoicePrint IME) R1

## 项目概述

构建一个双引擎驱动的个人化语音输入法：通用 ASR 引擎提供粗糙草稿，个人声纹映射引擎提供精确命中。核心机制是**后台评分排名，直接输出唯一结果**——没有候选栏，没有"指认仪式"，用户按住语音键说话，松手出字。用户每次修正输入的行为本身就是学习信号，后台排名自动调整，下次直接出对的。所有组件纯开源、纯离线、小巧轻量。

## 痛点：现有输入法根本不学习

"OpenClaw"——目前 IT 圈子里很流行的词，但你绝对找不到任何一个输入法能拼对它、记住它。你今天手动输入一次，明天它还是不认识。所有主流输入法的"学习"都是假的：词频调整是全局统计，不是个人记忆；新词要等云端更新，不是你教了就会；更不可能把你的发音习惯和你的用词关联起来。

VoicePrint IME 解决的不是"语音识别准确率"的问题，而是**"输入法根本不学习"**的问题。声纹只是让学习多了一个维度——同样的发音，你改成了什么，它就记住什么，下次直接给你。

## 当前状态

全新项目，无现有代码。需从零搭建。开发者无移动端开发经验，需预留框架学习时间。

**产品定位**：语音识别输入的特征替代者。只做语音输入，替代的是用户按语音键之后发生的事。不做键盘输入、不做联想、不做传统词频——用户按语音键说话，你给他更好的结果。

**终极目标**：让用户流畅聊天，用着用着就对了。用户纠正一次，下次同样发音就直接出正确的词。没有候选栏让你挑，后台评分排名后直接输出唯一结果。从 30% 准确率爬到接近 100%，靠的是"纠错即训练"的持续学习——学习速度和收敛速度是产品的核心指标。

## 需求条目

### REQ-001：通用语音识别引擎 (ASR Engine)

**定位**：提供粗糙的文字结果，让用户有东西可改，而非面对空白。

**功能**：
1. 接收麦克风音频流，输出文字结果
2. 完全离线运行，不依赖网络
3. 支持中文识别

**技术选型**：Vosk 开源模型（离线、支持中文、模型体积小、可跨平台编译）

**输出格式**：一段文字字符串，如 `"我要去公司"`

**准确率要求**：30% 即可。其唯一价值是提供可修改的草稿。

**集成方式**：C++ 层编译为 `libasr`，接收音频流，返回文字结果。

### REQ-002：个人声纹映射引擎 (Private Mapper)

**定位**：产品的核心。存储用户的声纹-文字关联，后台评分排名，输出唯一结果。

**核心概念**：没有"绑定"，没有"候选栏"，只有"概率排名→直接输出"。一条声纹可以关联多组文字，每组文字有不同的选中次数。排名由三个维度决定：**声纹相似度 × 选中频率 × 语境匹配度**。排名第一的直接输出，用户看不到排名过程。一次纠正只是加 1，不是一锤定音。多个结果撞在一起，反复出现的权重才真正拉开差距，自然收敛。

**存储**：SQLite，表结构：

**映射表 (mappings)**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| embedding | BLOB | 声纹指纹向量（float32 数组） |
| text | TEXT | 关联的文字 |
| context_tag | TEXT | 语境标签（如 App 名称），可为空 |
| hit_count | INTEGER | 被选中次数，默认 1 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后选中时间 |

**匹配逻辑**：
1. 对当前语音提取声纹指纹向量
2. 获取当前语境标签（App 名称、对话场景等）
3. 与库中所有向量计算余弦相似度
4. 对命中候选计算加权得分：`score = similarity × hit_count × context_boost`
   - `context_boost`：候选的 context_tag 与当前语境匹配时 > 1（如 2.0），不匹配时 = 1.0
   - 语境匹配让同一发音在不同 App 里输出不同的词
5. 返回相似度超过阈值的所有候选，按 score 降序排列
6. 阈值需可调（默认值需实测校准）

**输出**：排名第一的文字直接输出。如果没有命中，返回空（由流水线控制器决定走 ASR 结果）。

**关键约束**：
- 所有计算 100% 本地完成，绝不上传
- 指纹提取 + 比对总延迟 < 100ms
- 映射库支持增删改查

**库的建立与维护**：

声纹提取本身是成熟技术，真正的难点在库怎么建、怎么用。

*存储不是问题*：
- 256 维声纹模型，每条特征码 1KB（256 × 4 字节）
- 1000 组 ≈ 1MB，5000 组 ≈ 5MB，相比手机存储完全可以忽略
- 文本标签更小，几千组不到 0.2MB
- 不需要考虑存储优化

*建库策略——简单直接*：
- 每次用户修正，新增一条记录（embedding + text + context_tag），hit_count = 1
- 同一文字可能有多条近似向量（不同时间、不同状态说的），这是正常的，不需要聚类合并
- 同一文字的多条向量各自独立参与比对，各自维护 hit_count
- 理由：5000 条向量暴力比对，现代手机 CPU 单线程毫秒级完成，不需要索引结构

*调用策略——暴力比对即可*：
- 全量余弦相似度计算，无需 HNSW 等向量索引
- 10000 条 256 维向量，单次比对 < 20ms，远低于 100ms 延迟要求
- 一般用户映射库不超过 10000 条，暴力比对完全够用

### REQ-003：流水线控制器 (Pipeline Controller)

**定位**：协调双引擎，输出唯一结果。

**流程**：
1. 接收音频流
2. 并行执行：
   - 任务 A：音频 → ASR Engine → 粗糙文字
   - 任务 B：音频 → 声纹指纹提取 → Private Mapper 查询 → 排名第一的文字
3. 输出决策：
   - Private Mapper 有命中 → 输出排名第一的文字
   - Private Mapper 无命中 → 输出 ASR 结果

**输出逻辑**：直接上屏唯一结果，没有候选栏。用户看到的体验和普通语音输入完全一样——说话，出字。区别在于出的字越来越对。

**语境的作用**：同一发音，不同 App 里输出不同。在微信里说"gong si"，输出"工司"；在邮件 App 里说同样的发音，输出"公司"。语境让输入法知道你在跟谁说话。

### REQ-004：实时学习——纠正动作识别是核心算法

**定位**：整个产品最关键的设计。声纹提取不是焦点，**纠正动作识别**才是核心算法。系统必须准确识别用户退了什么、改成了什么，并关联到对应的声纹指纹。

**纠正动作的识别逻辑（第一版简化）**：

只识别一种纠正模式：**用户对刚上屏的末尾词按退格，然后输入新词**。中间隔了几个词再退回去改的，不处理。

识别逻辑极简：
1. 语音识别完成，上屏文字如 `"我要去公司"`
2. 系统监听退格键：如果退格发生在上屏文字的尾部区域（删的是最后几个字），记录删除内容
3. 用户输入新词，如 `"工司"`
4. 系统识别：尾部"公司"被替换为"工司"，关联当前声纹指纹
5. 写入映射库

**不处理的场景**：
- 用户隔了几个词再退回去改（如说完"我要去公司开会明天下午三点"，退到中间改"公司"）→ 不识别，不学习
- 用户直接按退格键删除（没有输入新词替代）→ 不识别，不学习
- 用户长按选中替换 → 第一版不处理

**为什么这样简化**：
- 不需要 diff 算法，不需要读整个输入框内容
- 只监听退格键发生在尾部 + 后续有新输入 = 纠正动作
- 覆盖最常见的纠正场景（说完发现最后一个词不对）
- 复杂纠正场景留给后续版本，或由 REQ-005 主动学习入口补充

**识别流程**：
```
上屏文字: "我要去公司开会"
用户操作: 退格×2 → 输入"工司"
最终文字: "我要去工司开会"
diff 结果: "公司" → "工司"，位置 index=3
关联声纹: 该位置对应的声纹指纹片段
写入映射: (声纹指纹, "工司", context_tag, hit_count=1)
```

**学习场景**：

**场景一：用户修正了 ASR 结果**
1. ASR 输出"公司"，用户退格删掉，输入"工司"
2. 系统识别纠正动作，将对应声纹指纹与"工司"写入映射库，hit_count = 1
3. 下次同样发音，Private Mapper 命中"工司"，直接输出"工司"

**场景二：用户修正了 Private Mapper 的结果**
1. Private Mapper 输出"工司"，但用户这次想要"公祠"，退格删掉，输入"公祠"
2. 系统将对应声纹指纹与"公祠"写入映射库，hit_count = 1
3. 下次同样发音，后台排名"工司(hit=5)" vs "公祠(hit=1)"，输出"工司"
4. 如果用户再次把"工司"改成"公祠"，"公祠"的 hit_count 递增，最终排名超过"工司"

**场景三：语境分化**
1. 微信里说"gong si"，ASR 输出"公司"，用户改成"工司"，写入映射库 context_tag=微信
2. 邮件里说"gong si"，ASR 输出"公司"，用户没改，系统将声纹指纹与"公司"写入映射库 context_tag=邮件
3. 下次微信里说"gong si"，context_boost 生效，输出"工司"；邮件里说，输出"公司"

**核心算法的难点**：

纠正动作识别不难，难的是**把纠正的词关联到声纹指纹的对应片段**。这取决于 REQ-001 中 ASR 是否提供词级时间对齐：
- 如果 Vosk 提供词级时间戳 → 可以精确提取被替换词的声纹片段 → 完美关联
- 如果 Vosk 不提供 → 只能用整句声纹 → 纠正"公司"时关联的是整句话的声纹，下次说"我要去公司开会"才能命中，单独说"公司"无法命中

**发音相似度过滤（否决机制）**：

纠正后的词和上屏词的发音必须有相关性。用户把"公司"改成"工司"，拼音相近，合理。改成"特国四"，发音完全不同，不合理——系统不应学习这种映射。

过滤逻辑：
1. 取上屏词的拼音序列，如"公司" → `gong si`
2. 取纠正词的拼音序列，如"工司" → `gong si`（完全匹配，通过）
3. 如"特国四" → `te guo si`（与 `gong si` 无相似性，否决）
4. 计算拼音序列的编辑距离或声母韵母相似度，超过阈值才写入映射库

**为什么需要这个否决**：
- 用户退格后输入的新词，可能不是在纠正语音识别错误，而是在追加新内容
- 例如：说完"我要去公司"，退格删掉"公司"，然后输入"回家"——这不是纠正，是改主意
- 发音相似度过滤能区分"纠正识别错误"和"改主意"，前者学习，后者不学习

**技术实现**：
- 汉字转拼音：开源库如 `pypinyin`（Python 原型）或 Android 内置拼音映射
- 相似度计算：声母+韵母逐位比对，或拼音序列编辑距离
- 阈值需实测校准：完全相同（0 距离）一定通过，差一个声母/韵母可能通过，完全不同一定否决

**场景四：用户修正自己的习惯（高权重信号）**
1. 用户在微信里一直说"gong si"输出"工司"，hit_count 已经到 20
2. 某天用户主动把"工司"改成"公司"——这是修正自己的习惯，不是随机波动
3. 系统识别：同一声纹 + 同一语境下，用户推翻了历史首选
4. "公司"的 hit_count 不是 +1，而是加一个更大的权重（如 +5 或 ×2），因为这是有意识的修正
5. 如果用户连续两次把"工司"改成"公司"，"公司"的权重会快速追上

**如何识别"修正习惯"vs"随机波动"**：
- 用户改的是当前语境下的历史首选词（hit_count 最高的那个）→ 修正习惯，高权重
- 用户改的不是首选词 → 普通纠正，正常 +1
- 上下语境是判断依据：同一声纹 + 同一语境 + 改的是历史首选 = 修正习惯

**权重策略**：
- 普通纠正：hit_count += 1
- 修正习惯：hit_count += N（N > 1，需实测校准，建议 3-5）
- 逻辑：推翻历史首选需要更强的信号，一次修正习惯应该抵得上几次普通选择

**核心认知：算法是调出来的，不是写出来的**

声纹匹配阈值、发音相似度阈值、修正习惯权重倍数、context_boost 值——这些参数没有理论最优解，只能用真实数据喂进去调。原型阶段的首要目标不是把参数调对，而是**搭好可调的框架**，让每个参数都能快速修改、快速验证。真正的算法逻辑是在调试中形成的。
- 纠正动作识别是核心算法，声纹提取不是
- 退格键是纠正的核心信号，有迹可循
- 没有候选栏，后台排名后直接输出唯一结果
- 没有绑定，只有概率排名
- 一次纠正只是加 1，不是一锤定音
- 多个结果撞在一起，反复出现的权重才真正拉开差距，自然收敛
- 同一声纹可以关联任意多组文字，各自独立计数
- 语境是第三维度：同一发音在不同 App 里可以有不同的最佳答案
- 学习是隐式的、持续的、无需用户额外操作的

### REQ-005：主动学习入口

**定位**：让用户主动补充声纹映射，加速冷启动，同时支撑"无屏幕识别调取"场景。

**交互流程**：
1. 用户进入"学习我的发音"页面
2. 系统展示一个文字（用户自输入或从常用词库选），如"OpenClaw"
3. 用户对着麦克风说这个词
4. 系统提取声纹指纹，存入映射库：(embedding, "OpenClaw", context_tag, hit_count=1)
5. 可选：让用户多说几遍，存入多条向量，提高后续命中率

**价值**：
1. **加速冷启动**：用户提前录入常用专有名词、人名、术语，不用等纠正才学习
2. **无屏幕识别调取**：录入后，用户说这个词就直接输出，不需要 ASR 中转。这是纯声纹驱动的输入，第一版就能做到
3. **补充纠正机制的盲区**：有些词用户永远不会通过"纠正"来教（比如人名、专业术语），主动录入是唯一的入口

**与纠正学习的关系**：主动录入和纠正学习写入同一张映射表，排名逻辑相同。主动录入只是多了一个入口，不是替代。

### REQ-006：个人词库管理

**定位**：让用户查看和管理已积累的映射数据。

**功能**：
1. 列表展示所有映射：文字 + 语境标签 + 选中次数 + 最后选中时间
2. 编辑：修改文字或语境标签
3. 删除：移除映射
4. 搜索：按文字内容搜索
5. 合并：将两条相似映射合并（同一文字、相近声纹），hit_count 叠加

### REQ-006：语境感知模块

**定位**：为排名提供第三个维度——语境。同一发音在不同语境下，最佳答案不同。

**语境信号来源**（按可靠性排序）：

1. **当前 App 名称**：输入法能感知当前输入目标 App（Android 可靠，iOS 受限）
2. **对话上下文**：输入框中已有的文字（如果输入法能读取），提供语义线索
3. **时间段**：工作时段 vs 休闲时段，粗粒度但零成本
4. **用户手动标签**：最可靠但需要用户操作，作为降级方案

**实现策略**：
- Android：直接获取当前 App 包名，映射为语境标签（如 `com.tencent.mm` → `微信`）
- iOS：暂不做
- 语境标签存储为字符串，匹配时做精确比较
- `context_boost` 默认值 2.0（语境匹配的候选得分翻倍），需实测校准

**降级方案**：如果所有自动信号都不可用，语境维度退化为不生效（即排名退回二维：相似度 × 频率），不影响核心功能。

### REQ-007：声纹指纹提取模块

**定位**：从音频中提取固定维度的特征向量。这是技术可行性的关键验证点。

**要求**：
1. 输入：一段音频（PCM，16kHz，单声道）
2. 输出：固定长度的 float32 向量（维度需实测确定，建议 128 或 256）
3. 提取延迟 < 50ms
4. 同一人说同一词，向量在特征空间中距离近
5. 不同人/不同词，向量在特征空间中距离远

**技术选型**（按优先级评估）：
1. **MFCC + 统计特征**：最轻量，零依赖，但区分度需实测验证。如果够用，这是最优解
2. **Vosk 内部特征**：如果 Vosk 暴露了中间层声学特征，可复用，避免引入额外模型
3. **开源轻量声纹模型**：如 SpeechBrain 的 ECAPA-TDNN 精简版，准确但模型体积和推理开销较大

**选型原则**：小巧优先。先验证方案 1 是否满足区分度要求，不够再升级到方案 2、3。

## 技术约束

1. **纯开源**：所有组件必须使用开源方案，不引入任何非开源组件
2. **小巧**：模型体积、内存占用、包大小均需控制。目标：安装包 < 100MB（含 ASR 模型），运行时内存 < 200MB
3. **跨平台核心层**：双引擎 + 流水线控制器用 C++ 实现，Android 通过 JNI 调用。第一版只做 Android，不上架 Play Store，侧载安装。iOS 版暂不做——苹果必须上架 App Store，审核麻烦
4. **100% 离线**：所有音频处理、指纹提取、映射匹配在本地完成，绝不上传任何声音与文本数据
5. **输入法进程约束**：系统可能随时杀掉输入法进程，需做好状态持久化（映射库写入 SQLite 即满足）
6. **音频格式**：PCM 16bit 16kHz 单声道（移动端标准）
7. **存储可忽略**：256 维声纹模型下，每条映射约 1KB，10000 组约 10MB，一般用户不超此数，无需存储优化或向量索引
8. **开发者无移动端经验**：需预留 Android InputMethodService 框架学习时间。建议开发路径：PC Python 原型验证核心逻辑 → Android 先跑通

## 验收标准

1. 在 PC 上（Python 原型）完成双引擎流水线演示：
   - 说话 → 输出 ASR 粗糙文字
   - 用户修正文字 → 系统自动学习
   - 再次说同样的话 → 直接输出修正后的文字（不是候选栏，是唯一结果）
   - 反复修正同一发音的同一词 → 该词排名持续靠前，输出越来越稳定
   - 切换语境标签 → 同一发音输出不同的词
2. 声纹指纹提取 + 映射查询总延迟 < 100ms（1000 条映射库）
3. 所有数据本地存储，无网络请求
4. 个人词库可查看、编辑、删除
5. 所有组件为开源方案
6. **收敛速度**：同一用户对同一词，修正 1 次后，下次同样发音直接输出修正后的词。这是产品的核心指标——纠正一次就要生效

## 已知风险

1. **声纹鲁棒性**：同一人在不同环境/情绪/语速下，声纹特征会有偏移。阈值设定需大量实测校准，不存在"一次设对"的可能
2. **冷启动体验**：初始映射库为空，用户需耐心修正。但修正本身就是正常使用，不增加额外负担
3. **iOS 语境标签获取**：iOS 版暂不做，此条暂时搁置
4. **MFCC 区分度**：最轻量方案可能区分度不足，需在原型阶段尽早验证。若不够，需引入更重的模型，与"小巧"约束产生张力
5. **修正检测的可靠性**：输入法感知用户修改上屏文字的能力，在不同平台、不同 App 中行为可能不一致
6. **唯一输出的代价**：没有候选栏意味着排名第二的词没有出口，用户只能通过再次修正来纠正。如果排名逻辑有偏差，用户体验会变差（比给候选栏更差）

## 外部评估校准

以下对照 DeepSeek 的难度分析（1-C），标注差异及理由：

**认同的：**
- 难度排序：平台框架(4) < ASR集成(3) < 流水线(5) < 声纹映射(7)——合理
- "不是发明新算法，是创造性嫁接"——准确，本项目的核心创新在产品机制而非算法突破
- 第二阶段（声纹映射）是成败关键——同意

**声纹映射难度偏高估（DeepSeek 7/10 → 实际约 4-5/10）：**
- DeepSeek 强调"调命中阈值"是最大难点，区分"同一个词"和"发音有点像"
- 但我们的设计是后台排名直接输出，阈值设低一点，顶多排名里多几个候选，第一名还是对的
- 容错空间大：误判的代价是排名里多了噪声项，不影响第一名输出
- 存储和性能也不是问题：5000 组才 5MB，暴力比对毫秒级，不需要索引结构
- 真正的难点不在声纹本身，而在"纠错即训练"的工程实现（修正检测、声纹时序关联）

**"纠错即训练"难度被低估（DeepSeek 5/10 → 实际约 6-7/10）：**
- DeepSeek 把流水线归为"工程逻辑活，不难但易出错"
- 但"检测用户修正行为"这件事远比看起来复杂：输入法上屏后，用户可能退格、可能选中替换、可能直接重输，不同 App 的输入框行为不一致
- 修正行为必须关联到当前会话缓存的声纹指纹，这个时序窗口的维护有工程复杂度
- 语境感知模块（获取 App 名称、对话上下文）是 DeepSeek 完全没提到的额外工作量

**时间估计偏乐观：**
- DeepSeek 估算单人 15-30 天做第一阶段（平台框架 + ASR 集成）
- 如果开发者对 Android InputMethodService 或 iOS UIInputViewController 不熟悉，光理解框架生命周期就要一周
- 建议第一阶段按 30-45 天估算，留出踩坑余量

**缺失的维度：**
- DeepSeek 完全没提语境感知，这是我们的第三排名维度
- DeepSeek 没有分析"小巧"约束带来的选型限制（纯开源、安装包 < 100MB），这会排除一些技术上更优但体积过大的方案
