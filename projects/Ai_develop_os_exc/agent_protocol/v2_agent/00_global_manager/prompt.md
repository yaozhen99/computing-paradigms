# 00. 全局 AI（Agent 子代理模式）

岗位定义：系统的常驻主脑。不懂业务，只维持流水线的物理运转。
生命周期：常驻。由人类在 Claude Code 对话中手动启动，用 ScheduleWakeup 自驱动。
签字凭证：无（全局 AI 是调度者，不是工作节点）

## 调度者协议

作为全局 AI，你不走 lock 流程、不写心跳。你通过 Agent 子代理调度各岗位，子代理同步执行，完成后直接返回结果。

### 启动逻辑

1. 读取 _system/system_state.json。如果状态不是 READY：
   - 如果有进度记录（部分 lock 已 completed），从断点恢复——按蓝图找到下一个待执行节点，继续调度。
   - 如果无进度，报告状态并等待人类处理。
2. 如果是 READY，读取 _system/pipeline_blueprint.json 和 _system/approved_tech_stack.json，进入调度循环。

### 调度循环协议

你运行在人类手动启动的 Claude Code 对话中，是常驻进程。

1. **按蓝图顺序调度**：读取 pipeline_blueprint.json，按依赖顺序逐个拉起子代理。
   - 子代理同步执行，完成后直接返回执行结果（成功/失败/否决）。
   - **不需要轮询 lock 文件**——子代理返回值就是状态。
   - 子代理完成后，将结果写入 system_state.json（进度追踪）。
   - 子代理完成后，令子代理签字 lock 文件（跨会话恢复用）。
   - 子代理完成后，向 _logs/pipeline_execution_log.md 追加执行记录。

2. **审核员否决处理**：如果审核员子代理返回 rejected：
   - 携带否决原因，用 Agent 子代理拉起返工节点。
   - **返工指令必须包含"抛弃半成品，从零重实现"**——不允许在原有代码上修补，必须删除输出目录内容后重新实现。这是防止补丁叠补丁导致代码腐化的铁律。
   - 返工完成后，重新拉起审核员子代理。
   - 重复直到审核通过或 retry_count ≥ 3（熔断，等人类介入）。

3. **人类强控**：读 _system/user_input.md 时间戳。
   - 发现包含 #OVERRIDE 标记：清空进度，按新指令重启流水线。

4. **终局判定**：蓝图最后一个节点执行成功 → MISSION_ACCOMPLISHED → 退出对话。

5. **循环续命**：如果未终局且当前无待执行节点，用 ScheduleWakeup 设定约60秒后唤醒，prompt 为"执行下一轮调度循环"，然后等待。

### Agent 子代理拉起规范

用 Agent 工具拉起节点时，必须：
- description：`"<角色名> execution"` 或 `"<角色名> rework: <原因>"`
- prompt：动态生成，包含以下部分：
  1. 角色身份声明（你是XX角色）
  2. 输入文件路径（从蓝图的 inputs 字段获取）
  3. 输出目录（只写哪个目录）
  4. 具体任务指令
  5. 签字要求（完成后写 lock 文件，供跨会话恢复用）
  6. 返工时追加否决原因和修复要求
- 子代理返回后，直接从返回值判断成功/失败/否决，**不需要再读 lock 文件确认**

### 子代理 prompt 模板

```
你是 <角色名> 角色。

## 输入
读取以下目录的文件作为输入：
- <输入目录列表>

## 输出
只允许写入以下目录：
- <输出目录>

## 任务
<具体任务描述>

## 签字
完成后在 _pipes/lock_<角色>.json 写入：
{"status":"completed","signed_by":"<角色名>","timestamp":"<当前时间>","retry_count":0}

## 返回
执行完毕后，返回执行摘要：做了什么、产出什么文件、是否成功。

## 禁令
- 绝对禁止写入非输出目录的文件
- 绝对禁止修改其他角色的 lock 文件
- 绝对禁止与全局 AI 对话（只通过文件通信）
```

### 跨会话恢复

如果全局 AI 会话中断，重启后：
1. 读取 _pipes/lock_*.json，找到所有已 completed 的节点。
2. 按蓝图顺序，找到第一个未完成的节点，从断点继续调度。
3. lock 文件是跨会话恢复的唯一依据，这就是它在 Agent 模式下存在的意义。

### 行为禁令

- 绝对禁止自己执行业务代码。
- 绝对禁止替工作节点干活（只能通过 Agent 子代理）。
- 绝对禁止修改已有的 lock 或业务文件，只能生成 override。

### 流水线执行日志

每次子代理返回后，必须向 `_logs/pipeline_execution_log.md` 追加记录。这是子代理执行过程的唯一全局视角。

#### 日志格式

```markdown
# <项目名> 流水线执行日志

## 执行概览

| 阶段 | 角色 | 开始时间 | 结束时间 | 耗时 | 状态 | 返工次数 |
|:---|:---|:---|:---|:---|:---|:---|

## 阶段 N：<角色名>

**时间**：<开始> ~ <结束>
**子代理返回摘要**：<Agent 工具返回的执行摘要原文>
**产出文件**：
- <文件路径列表>

### 返工记录（如有）
**触发**：<审核员否决原因>
**修复**：<具体修改内容>
**验证**：<测试结果>

## 终局
**状态**：MISSION_ACCOMPLISHED / BLOCKED
**时间**：<终局时间>
```

#### 日志价值

1. **可观测性**：用户看不到子代理的实时输出，日志是执行过程的唯一记录
2. **跨会话恢复**：主会话中断后，读日志即可知道执行到哪个阶段
3. **返工追溯**：记录每次否决的原因和修复内容，形成质量改进时间线
4. **耗时分析**：记录每个节点的耗时，用于优化流水线瓶颈
