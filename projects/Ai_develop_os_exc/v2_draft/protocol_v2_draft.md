# 纯会话驱动 AI 协作协议 V2 (初稿)

> 适配 Claude Code 的 Agent 子代理机制，从"多进程文件协调"转向"主会话+子代理协作"。

## V1 的核心问题

V1 协议基于一个假设：每个岗位是一个独立的 Claude Code 进程，通过文件系统协调。
实际执行中暴露的矛盾：

1. `claude -p` 新窗口启动不可靠，节点经常跑不起来
2. 文件中转（心跳、lock、override）在同步调度场景下是冗余的
3. 感知循环（定时轮询）没有原生支持，只能手动轮询
4. 每个节点"启动→读文件→写文件→退出"的生命周期，在 Agent 子代理中天然不成立——子代理是一次性执行完就返回结果

## V2 核心思路

**一个主会话 = 全局 AI，子代理 = 工作节点。**

主会话持有完整上下文，按蓝图顺序用 Agent 工具拉起子代理执行任务。
子代理之间不直接通信，但主会话可以传递关键上下文（如前序节点的产出摘要）。

## 角色定义

### 主会话（全局 AI / 调度者）

- 读取蓝图，按依赖顺序拉起子代理
- 不做业务工作，只调度
- 接收子代理返回的执行摘要，决定下一步
- 审核员否决时，携带 override 原因重新拉起返工子代理

### 子代理（工作节点）

- 每个子代理 = 一个岗位的一次执行
- 接收：任务指令 + 输入文件路径 + 产出文件路径 + 协议约束
- 返回：执行摘要（做了什么、产出什么、是否成功）
- 遵循：防御性写入（.tmp → 拷贝覆盖）、目标先行、手术式修改

## 简化掉的东西

| V1 机制 | V2 处理方式 |
|:---|:---|
| 心跳文件 | 删除。子代理同步执行，主会话直接等待返回 |
| lock 文件 | 删除。主会话跟踪状态，不需要文件锁 |
| override 文件 | 删除。审核子代理返回 rejected + 原因，主会话直接拉起返工子代理 |
| 感知循环 | 删除。主会话按蓝图顺序串行/并行调度，无需轮询 |
| system_state.json | 简化为蓝图完成进度，主会话内存中维护 |
| _heartbeats/ 目录 | 删除 |
| _pipes/ 目录 | 删除 |
| _scripts/ 目录 | 删除（不再需要启动脚本） |
| _prompts_active/ 目录 | 删除。prompt 由主会话在拉起子代理时直接写入 |

## 保留的东西

| V1 机制 | 保留原因 |
|:---|:---|
| 防御性写入（.tmp → 覆盖） | 防止写入中断导致文件损坏，仍然必要 |
| 蓝图（pipeline_blueprint.json） | 定义岗位依赖关系和流转顺序，调度依据 |
| 技术栈裁决（approved_tech_stack.json） | 人类审批卡口，不可省略 |
| _shared/ 共享协议 | node_protocol.md、AGENTS.md、CLAUDE.md 仍需子代理遵守 |
| _logs/ 经验沉淀 | Hermes 协议仍有价值 |
| _skills/ 技能索引 | 跨会话知识积累 |
| 文档模板 | 产出格式一致性仍需保证 |
| 审核员一票否决 | 质量卡口不可省略 |
| 人类强控（#OVERRIDE） | 保留，但改为直接在主会话中下达指令 |

## 执行流程

### 相位一：创世（任务师）

与 V1 相同，任务师独立执行：
1. 读取需求 → 生成技术选型提案 → 等待人类审批
2. 读取审批结果 → 创建目录结构 + 蓝图 → 退出

产出物：目录结构 + approved_tech_stack.json + pipeline_blueprint.json + _shared/

### 相位二：运转（主会话调度）

主会话启动后：

```
1. 读取 pipeline_blueprint.json，构建依赖图
2. 找到所有无前置依赖的节点，并行拉起子代理
3. 等待子代理返回：
   a. 成功 → 标记该节点完成，检查是否有新节点解锁
   b. 失败 → 重试（最多3次），超过则暂停等人类介入
   c. 审核员否决 → 携带原因拉起返工子代理
4. 所有节点完成 → MISSION_ACCOMPLISHED
```

### 子代理调用规范

主会话拉起子代理时，prompt 必须包含：

```
1. 身份声明：你是 XX 项目的 [角色] 节点
2. 工作目录：绝对路径
3. 节点协议：遵守 _shared/node_protocol.md（防御性写入、目标先行等）
4. 输入文件：绝对路径列表
5. 产出文件：绝对路径列表 + 格式模板路径
6. 业务指令：具体执行步骤
7. 返回要求：执行摘要（做了什么、产出什么、是否成功）
```

### 并行规则

- 无依赖的节点可以并行拉起（如 dba + uiux）
- 有依赖的节点必须等前置完成
- 审核员必须串行（等所有输入就绪）

## 与 V1 的对比

| 维度 | V1 | V2 |
|:---|:---|:---|
| 进程模型 | 多个独立 Claude Code 窗口 | 一个主会话 + 子代理 |
| 协调机制 | 文件系统（心跳/lock/override） | 会话上下文（Agent 返回值） |
| 调度方式 | 感知循环（定时轮询） | 事件驱动（子代理完成后触发） |
| 状态存储 | 文件（system_state.json） | 主会话内存 + 蓝图文件 |
| 容错 | 心跳超时→斩首→重启 | 子代理失败→重试→熔断 |
| 人类介入 | #OVERRIDE 文件 | 直接在主会话中对话 |
| 启动脚本 | .sh/.bat | 不需要 |
| prompt 存储 | _prompts_active/ | 主会话动态生成 |

## V1 实战暴露的设计缺陷

### 缺陷一：岗位裁剪误判——release 不等于 CI/CD

vault-cli 项目裁剪掉了 13_release，理由是"v1 本地工具，无需正式发布流程"。
但审核通过后，产出的只是散落在 03_source/backend/ 里的源码文件，没有一个节点负责：
- 把代码组装成可交付物（pyproject.toml / setup.py / 入口脚本）
- 版本号打标
- 打包为 pip installable 或独立可执行文件

**"无需正式发布流程" ≠ "无需组装成可运行的整体"。**

release 岗位实际承担两个职责：
1. **合成职责**（不可裁剪）：将源码树转化为可安装/可运行的交付物
2. **发布职责**（可裁剪）：CI/CD、容器化、版本分发、changelog

任务师裁剪时把两者混为一谈，导致项目"代码写完了但跑不起来"。

**V2 修正**：蓝图模板中，release 岗位的合成职责拆为独立阶段 `integration`，不可裁剪。发布职责保留为可选的 `release` 阶段。

### 缺陷二：测试节点依赖声明不完整

V1 蓝图中 tester_write 依赖 backend，tester_run 依赖 tester_write。
但 tester_write 还需要读取 api_contracts.md（来自架构师），这个隐含依赖没有在蓝图中声明。
在多进程模式下可能导致 tester_write 在架构师完成前就被拉起。

**V2 修正**：蓝图依赖必须显式声明所有输入来源，不允许隐含依赖。

## 流水线执行日志

主会话在每次子代理返回后，必须向 `_logs/pipeline_execution_log.md` 追加记录。

### 日志格式

```markdown
# <项目名> 流水线执行日志

## 执行概览

| 阶段 | 角色 | 开始时间 | 结束时间 | 耗时 | 状态 | 返工次数 |
|:---|:---|:---|:---|:---|:---|:---|

## 阶段 N：<角色名>

**时间**：<开始> ~ <结束>
**子代理返回摘要**：<Agent 工具返回的执行摘要原文>
**产出文件**：
- <文件路径列表>

### 返工记录（如有）
**触发**：<审核员否决原因>
**修复**：<具体修改内容>
**验证**：<测试结果>

## 终局
**状态**：MISSION_ACCOMPLISHED / BLOCKED
**时间**：<终局时间>
```

### 日志价值

1. **可观测性**：用户在主会话中看不到子代理的实时输出，日志是唯一的执行过程记录
2. **跨会话恢复**：主会话中断后，读日志即可知道执行到哪个阶段、哪些节点已完成
3. **返工追溯**：记录每次否决的原因和修复内容，形成质量改进的时间线
4. **耗时分析**：记录每个节点的耗时，用于优化流水线瓶颈

### 与心跳文件的关系

V1 的心跳文件（_heartbeats/hb_*.md）是节点自己写的实时状态，V2 的执行日志是主会话写的全局视角。
两者互补：心跳记录节点的微观动作（"第5步：创建 crypto/service.py"），日志记录宏观流转（"阶段3：后端开发，耗时60min，返工1次"）。

V2 Agent 模式下心跳可选保留（子代理自行写入），但执行日志必须由主会话写入。

## 未解决的问题（待讨论）

1. **跨会话恢复**：如果主会话中断，如何恢复进度？V1 靠文件系统天然支持，V2 需要额外机制
2. **并行子代理的冲突**：两个子代理同时写同一目录下的文件，需要主会话协调
3. **子代理上下文窗口**：大型项目的代码树可能超出子代理的读取能力
4. **任务师是否也需要改为子代理模式**：目前任务师仍是独立窗口执行
5. **岗位裁剪的安全边界**：如何防止任务师误裁不可裁剪的职责？是否需要"最小必选岗位集"硬约束？
