# CiviBBS 迭代评审方法论

> 来源：Tony × 多模型交叉评审实践（wait_for_mail_arrival / send_shutdown_to_topo_unit）
> 用途：开发岗、审核岗、所有参与代码迭代的 AI 和人类
> 核心原则：**质量不取决于单个模型的能力上限，而取决于迭代 + 交叉审核的机制**

---

## 一、为什么需要迭代评审

单次评审无论多强，都有盲区。原因：

1. **建造者和评审者是两种认知模式** — 写代码时关注"怎么解决问题"，评审时关注"这里还能出什么问题"。同一个模型切换角色后能看到不同层面的问题。
2. **质量基线抬升后，低级问题消失，高级问题才浮出水面** — P0 修完后才能看到 P1，P1 修完后才能看到 P2。就像一栋楼还在漏水时，你不会挑剔墙漆色差。
3. **不同模型有不同的盲区分布** — Opus 强于架构/契约，GPT5.4 强于测试设计，GLM5.1 强于细节/可行性，DeepSeek V4 强于执行+元评审。交叉使用比单模型多轮更有效。

**实证：** send_shutdown_to_topo_unit 经历三轮评审（Opus→GLM5.1→DeepSeek V4），从 6 分提升到 8.5 分，每轮发现的问题维度不同但都在缩小残差。

---

## 二、迭代收敛公式

```
最终质量 = 迭代次数 × 视角多样性 × 基线提升
```

- **迭代次数**：不是重复做同一件事，而是每轮站在前一轮成果上
- **视角多样性**：不同模型、不同角色（建造者/评审者）、不同维度（功能/安全/测试/契约）
- **基线提升**：每轮修完 P0，下一轮才能看到 P1；修完 P1，才能看到 P2

---

## 三、评审维度清单

每轮评审应覆盖以下维度，确保不遗漏：

| 维度 | 关注点 | 典型问题 |
|------|--------|----------|
| **契约一致性** | definition ↔ execute ↔ triggers 三者是否对齐 | 幽灵错误码（声明了没实现）、输出字段形状不一致 |
| **安全性** | 路径穿越、空字节注入、类型注入（bool→int） | unit_id 含 `../`、`isinstance(True, int)==True` |
| **错误处理** | 异常捕获范围、错误码粒度、降级策略 | `except Exception` 过宽、静默降级违反规范 |
| **测试真实性** | 触发器是否真的能测到想测的东西 | Path 对象测的是字符串、mock 框架不支持、chmod 在 Windows 无效 |
| **边界语义** | 零值、空值、极端参数的行为是否明确 | timeout=0 是"不超时"还是"立即超时"？多文件返回哪个？ |
| **跨平台** | Windows/Linux 行为差异 | 权限模型、路径分隔符、保留文件名 |
| **可调试性** | 失败时信息是否足够定位问题 | unit_id 失败时是否回显、是否有完整堆栈 |

## 四、迭代流程

```
第1轮：开发 + 自审
  ├── 开发岗按 anti-patterns.md 写代码
  ├── 自审：切换评审者视角，按维度清单逐项检查
  └── 产出：v1.0 + 自审报告

第2轮：交叉审核
  ├── 不同模型/角色审核
  ├── 对照 anti-patterns.md + 维度清单
  └── 产出：审核意见 + 修复后版本

第3轮：收敛审核
  ├── 审核前一轮修复是否引入新问题
  ├── 检查测试真实性（触发器是否真的能跑）
  └── 产出：最终版本 + 审核通过

第N轮：按需继续（见停止条件）
```

## 五、停止条件（防止过度优化）

满足以下**任一**条件即停止迭代：

| 条件 | 说明 |
|------|------|
| 评分收敛到 9+ 且连续两轮无新 P0/P1 | 质量已足够高 |
| 单轮评审只发现 P3（风格/偏好） | 剩余问题不值得迭代成本 |
| 迭代超过 5 轮 | 强制停止，当前质量已足够 |
| 修复引入的新问题连续两轮比修掉的更多 | 收益递减，停止 |

**优先级定义：**
- **P0**：阻塞性，功能不可用或安全漏洞（如幽灵错误码、路径穿越）
- **P1**：应修复，语义错误或测试不可靠（如 E_TIMEOUT 声明不实现、测试假阳性）
- **P2**：优化项，不影响核心功能（如日志级别、格式统一）
- **P3**：风格偏好，纯主观（如变量命名风格）

## 六、评审者角色切换技巧

同一个模型/人做自审时，必须**显式切换角色**：

**建造者模式（写代码时）：**
- "怎么把已知问题解决掉？"
- 关注功能完整性、错误码覆盖、代码结构

**评审者模式（审代码时）：**
- "这真的能工作吗？"
- 关注：mock 框架真的支持吗？Path 对象真测到了吗？Windows 上 chmod 有效吗？
- 假设所有代码都有问题，逐行质疑

**元评审模式（审评审时）：**
- "评审本身有没有漏判/误判？"
- 关注：优先级划分是否合理？有没有吹毛求疵？有没有错评？

## 七、实战案例

### send_shutdown_to_topo_unit 三轮评审链

| 轮次 | 评审者 | 发现 | 修复 |
|------|--------|------|------|
| 第1轮 | Claude Opus | P0: E_TIMEOUT 幽灵码、路径穿越漏洞、not_found 语义错误 | 实现超时、加路径穿越防护、is_dir() 预检 |
| 第2轮 | GLM5.1 | P1: Path 对象测试假阳性、except Exception 过宽、mock 标注缺失 | 重新设计触发器、收窄异常捕获、加 mock_required |
| 第3轮 | DeepSeek V4 | P2: strip/空字节顺序、signal_path 类型标注、默认值冗余 | 调整校验顺序、补 nullable 标注、统一默认值处理 |

**关键发现：** GLM5.1 自己写的代码，自己评审时还能找出 7 个问题。不是能力不够，是角色切换改变了注意力分配。

### wait_for_mail_arrival 测试设计倒逼实现

| 阶段 | 做了什么 | 发现 |
|------|----------|------|
| GPT5.4 测试设计 | 从 3 个 trigger 扩展到 30 个测试矩阵 | .eml 目录被 glob 误匹配（实现 bug） |
| Opus 代码评审 | 审规范 vs 实现的契约一致性 | E_CANCELLED 幽灵码、inbox 不存在返回 E_TIMEOUT 而非 E_PATH_INVALID |
| GPT5.4 代码实现 | v1.0.3 修复 + 完整测试集 | _list_mail_files 只认 is_file()、_is_plain_int 排除 bool |

**关键发现：** 测试设计过程本身就能发现实现 bug，"先设计测试再写代码"不是空话。
