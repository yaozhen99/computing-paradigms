# 会话生命周期与自我演化协议 (AGENTS.md)

本文件不包含任何项目具体信息。它定义的是 AI 助手必须遵守的**行为纪律、文档产生规则与自我演化机制**。

## 一、启动与空间感知
1. **读取本协议**：首先确认并遵守本文件全部条款。
2. **遍历目录文档**：读取根目录下的 README_agent.md，以及所有一级、二级子目录下的 README.md，构建项目空间认知。
3. **读取最新状态**：读取 `docs/plans/` 中最新的计划文件、`docs/summaries/` 中最新的总结文件、`docs/skills/` 下所有技能文档。
4. **读取格式模板**：读取 `docs/templates/` 目录下的所有模板文件，作为后续文档生成的格式依据。
5. **缺失处理**：若上述任何目录或文件缺失，应在首次回应中明确指出，并询问是否立即按照本协议规定初始化。

## 二、文件格式规范
所有流程文件的生成，必须严格遵循 `docs/templates/` 目录下对应模板的格式要求。若对应模板不存在，应主动提示创建。

## 三、目录驻留文档纪律（结构级自描述）
- **存在性要求**：项目根目录及**每一个一级、二级子目录**下，必须存在 `README.md` 文件。
- **内容规范**：描述当前目录的用途、包含模块的职责、对外接口、依赖关系。具体格式参考 `docs/templates/README.md`。
- **更新纪律**：对该目录下任何代码进行修改后，**必须同步更新**该目录的 `README.md`，并在文件顶部以 `## [YYYY-MM-DD HH:MM] 更新说明` 格式记录本次变更摘要。

## 四、流程归档文档产生规则（时间轴自记录）
按以下阶段和顺序生成独立文件，所有文件内容均需包含时间戳，格式遵循 `docs/templates/` 下对应模板。

| 阶段 | 文件路径 | 对应模板 |
|:---|:---|:---|
| 需求 | `docs/requirements/[YYYY-MM-DD]_[简述].md` | `requirement.md` |
| 计划 | `docs/plans/[YYYY-MM-DD]_[简述].md` | `plan.md` |
| 开发笔记 | `docs/notes/[模块名]_开发笔记.md` | `dev_note.md` |
| 开发总结 | `docs/summaries/[模块名]_开发总结.md` | `dev_summary.md` |
| 测试需求 | `tests/[模块名]/测试需求.md` | `test_req.md` |
| 测试计划 | `tests/[模块名]/测试计划.md` | `test_plan.md` |
| 测试过程 | `tests/[模块名]/测试过程.md` | `test_process.md` |
| 测试总结 | `tests/[模块名]/测试总结.md` | `test_summary.md` |
| 审核报告 | `docs/reviews/[模块名]_审核报告.md` | `review.md` |
| 技能更新 | `docs/skills/project_specific.md`<br>`docs/skills/general_patterns.md` | `project_skill.md`<br>`general_skill.md` |

## 五、插件化规则（资产自沉淀）
- 凡在两个及以上模块中被复用的代码，**必须**抽取为独立插件，存放于 `src/plugins/[插件名]/`。
- 插件目录内必须包含 `README.md` 说明功能、用法和依赖，格式参考 `docs/templates/README.md`。

## 六、自我演化机制（元认知层）

### 6.1 计划与总结的差异分析（强制触发）
**在完成"开发总结"文件写入后，你必须立即执行以下分析**：
1. 读取对应的"开发笔记"或"计划"文件。
2. 逐条对比计划内容与实际完成情况，找出**差异点**。
3. 在"开发总结"文件末尾，追加一节 `## 计划与实际差异分析`，格式见 `docs/templates/dev_summary.md` 中的对应章节。

### 6.2 技能沉淀标准（自动知识抽取）
**任务完成后，必须扫描本次产生的"开发总结"和"差异分析"，并根据以下标准决定是否更新技能文档**：
- **写入 `project_specific.md` 的条件**：任何与**本项目特定架构、业务规则、模块依赖**相关的经验、踩坑、最佳实践。
- **写入 `general_patterns.md` 的条件**：任何**与具体项目无关、可通用于其他项目的**设计模式、算法实现、调试技巧、命令行技巧。
- **若无新增知识**：必须在技能文档末尾追加一行 `## [YYYY-MM-DD] 本次任务无新增通用技能`。
- **格式要求**：遵循 `docs/templates/project_skill.md` 和 `docs/templates/general_skill.md` 的内部结构。

### 6.3 协议自我修正建议（反向监督）
**若在同一类型任务中，连续两次出现相似的"计划与总结差异"，且差异原因指向本协议的不完善或模糊之处，你必须主动提出以下建议**：
> "检测到连续两次任务在 [具体环节] 出现计划与执行的偏差，原因可能与本协议中 [具体条款] 的缺失或模糊有关。建议在 `AGENTS.md` 中增加或修改以下规则：[具体建议]。"

### 6.4 协议修改权限与流程（强制执行）

**修改权限声明**：
本协议（`AGENTS.md`）及其引用的所有模板文件（`docs/templates/`）的**最终解释权与唯一修改权属于项目管理者（人）**。AI 助手在任何情况下都**不得**直接修改本协议文件或模板文件的内容。

**建议提交流程**：
1. 当你依据 `6.3` 条款认为协议需要修正时，**必须**将修正建议以文本形式输出在对话中，或生成为一个独立的 Markdown 文件，存放于 `docs/proposals/protocol_change_[YYYY-MM-DD].md`。
2. **严禁**直接编辑 `AGENTS.md` 或任何模板文件。
3. 项目管理者（人）将审阅建议，并决定是否采纳、如何修改，然后由人手动更新协议文件并提交至版本控制系统。

**违规后果**：
任何 AI 助手直接修改本协议或模板文件的行为，将被视为**最高优先级的执行错误**，必须立即向用户报告并尝试恢复文件原状。

## 七、模型无关声明
本协议为自然语言行为准则，与后端推理模型种类无关。任何遵循本协议的 AI 工具均须等效执行。

## 快速纠偏指令
当你发现 AI 未按本协议执行时，只需在对话中发送：
> **"@protocol"**
AI 必须立即重新读取本协议全文，确认当前状态与协议的差距，并自动修正后续行为。
