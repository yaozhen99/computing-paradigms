# PRD: StateKanban -- 基于状态看板的指令级开发引擎

> **迭代轮次**: 第二轮 -- 驱动循环 + Codex 接入
> **基线**: 第一轮已完成六大内核模块（kanban/message_bus/viewport/valve/registry/process），3987 行代码，203 项测试全部通过

## 1. 项目概述与目标

### 1.1 项目定位

StateKanban 是一个纯内存的 LLM 调度内核，将 LLM 降级为无状态算力单元，以状态看板为唯一真相源，通过视口切片控制上下文，以输出阀门保证写入安全。系统底座完全隔绝与操作系统的直接交互，内部进程通过消息总线高频对话，所有外部动作通过工具调用完成。

### 1.2 第二轮增量目标

第一轮交付了静态内核（六大模块），但 `statekanban run --intent "xxx"` 只做了初始化就停了——缺少将看板信号流转串联起来的驱动循环。本轮补完驱动引擎并接入 Codex，让系统能真正跑起来。

| 编号 | 增量目标 | 衡量标准 |
|------|----------|----------|
| G6 | 驱动循环可运行 | `statekanban run --intent "xxx"` 启动后自动执行 Coder→Reviewer→Tester→晶态固化→阀门写出的完整循环 |
| G7 | 碰撞收敛自动化 | 意图与否决信号碰撞后自动收敛，90% 在 3 轮内达成一致 |
| G8 | Codex 算力单元 | call_codex 工具注册到 ToolRegistry，Coder/Integrator 可调用 Codex 产出代码 |
| G9 | 适配层可切换 | CLI `--adapter` 选项在 mock/anthropic/cli/codex 之间切换，引擎代码无需修改 |
| G10 | 熔断保障 | 超过最大轮数后正确熔断并报告人类，不进入死循环 |

### 1.3 保留的核心目标（第一轮）

| 编号 | 目标 | 衡量标准 |
|------|------|----------|
| G1 | LLM 调用无状态化 | 每次调用 LLM 的输入完全由视口切片器生成，无会话历史依赖 |
| G2 | 状态可恢复 | 进程崩溃后从看板快照恢复，状态零丢失（100%） |
| G3 | 输出安全 | 所有物理写入经过阀门校验链，拦截率 95%+ |
| G4 | 意图收敛 | 碰撞循环 90% 在 3 轮内收敛 |
| G5 | 单机可用 | 作为可安装 Python 包，通过 CLI 启动并驱动开发任务 |

## 4. 用户故事 / 使用场景

### US-01：启动开发任务（第一轮）

> 作为开发者，我希望通过 CLI 命令启动一个开发任务（如"实现用户登录模块"），系统自动创建 Coder/Reviewer/Tester 进程，通过看板驱动代码生成、审核、测试的完整循环。

### US-02：意图碰撞与收敛（第一轮）

> 作为系统，当 Coder 提交代码意图与 Reviewer 的否决信号在流态区碰撞时，自动触发收敛循环，直到双方达成一致，产出进入晶态区。

### US-03：校验失败返工（第一轮）

> 作为系统，当晶态区产出通过输出阀门时，若语法检查或测试执行失败，错误信号回注流态区，触发 Coder 返工循环。

### US-04：崩溃恢复（第一轮）

> 作为开发者，当进程意外崩溃后，系统能从最近的看板快照完整恢复所有状态，继续未完成的任务。

### US-05：权限隔离（第一轮）

> 作为系统管理员，我希望每个进程角色只能调用其声明的工具子集，确保即使 LLM 输出异常指令也不会越权。

### US-06：视口控制（第一轮）

> 作为系统，每次调用 LLM 前自动裁剪上下文，确保 Coder 只看到与代码生成相关的信息，Reviewer 只看到需要审核的代码。

### US-07：快照与恢复（第一轮）

> 作为开发者，我希望手动触发看板快照并在需要时从快照恢复。

### US-08：扩展新角色（第一轮）

> 作为系统架构师，我希望通过工具注册表声明新角色的权限。

### US-09：驱动循环自动执行（第二轮新增）

> 作为开发者，我希望 `statekanban run --intent "实现X模块"` 启动后，系统自动驱动 Coder→Reviewer→Tester→Integrator 的完整循环，直到碰撞收敛、晶态固化、阀门写出，不需要我手动干预每一步。循环结束后，我看到结果摘要，知道产出文件在哪里。

### US-10：碰撞收敛自动化（第二轮新增）

> 作为系统，当 Coder 产出代码意图但 Reviewer 发出否决信号时，我自动进入收敛循环：Coder 看到否决原因后修改代码，Reviewer 再次审核。如果达成一致，产出进入晶态区；如果超过 10 轮仍未收敛，我熔断并报告人类。

### US-11：Codex 代码生成（第二轮新增）

> 作为 Coder 角色，我可以通过 call_codex 工具调用 OpenAI Codex 生成代码。Codex 看到我提供的 prompt 和上下文文件，产出代码片段，我将其作为意图信号写入流态区。

### US-12：Codex 权限隔离（第二轮新增）

> 作为 Reviewer 角色，我不应该能够调用 call_codex 工具。我只能使用 call_llm 进行审核判断。如果尝试调用 call_codex，系统应该拒绝并记录审计日志。

### US-13：适配层切换（第二轮新增）

> 作为开发者，我希望通过 `--adapter codex` 选择使用 Codex 作为算力单元，或通过 `--adapter mock` 在无 API Key 的环境下测试。引擎代码不需要为不同的适配层做任何修改。

### US-14：熔断保护（第二轮新增）

> 作为开发者，如果碰撞收敛循环超过 10 轮（或我指定的 `--max-rounds`），系统应该熔断并告诉我当前状态，而不是无限循环下去。

### US-15：LLM 返回值解析容错（第二轮新增）

> 作为系统，当 LLM 返回的 JSON 无法解析时，我不崩溃，而是将其作为错误信号回注流态区，让下一轮循环重试。

### US-16：详细循环日志（第二轮新增）

> 作为开发者，我使用 `--verbose` 选项时，能看到每轮循环的详细信息：哪个角色在处理、视口切片了哪些内容、LLM 返回了什么信号、碰撞状态如何。

## 5. 验收标准

### AC-01：核心引擎可用（第一轮，保留）

- [ ] StateKanban 内核可通过 `statekanban run --intent "实现X模块"` 启动
- [ ] 系统自动创建所需进程角色并启动消息总线

### AC-02：驱动循环可运行（第二轮新增）

- [ ] `statekanban run --intent "xxx"` 启动后自动执行驱动循环
- [ ] 循环执行 Coder→Reviewer→Tester→Integrator 角色调度顺序
- [ ] 每轮循环正确执行：信号读取→视口切片→LLM调用→返回值解析→信号回传→碰撞检测
- [ ] 收敛后正确触发：晶态区固化→输出阀门校验→物理写出
- [ ] 循环结束后输出结果摘要（收敛状态、产出文件、轮数统计）
- [ ] `--max-rounds N` 选项生效，默认值 10

### AC-03：碰撞收敛（第一轮 + 第二轮增强）

- [ ] 意图信号与否决信号在同一 target_id 上碰撞时自动触发收敛循环
- [ ] 碰撞循环 90% 在 3 轮内收敛（通过 Mock LLM 测试套件验证）
- [ ] 收敛条件：最新一轮没有新的否决信号，或否决信号被意图信号覆盖
- [ ] 未收敛时自动进入下一轮循环
- [ ] 超过最大轮数后正确熔断，报告人类
- [ ] 收敛过程有完整日志记录

### AC-04：LLM 返回值解析（第二轮新增）
