## 一、对话全景

### 时间线

| 阶段 | 谁 | 做了什么 | 耗时 |
|------|-----|----------|------|
| 1. 批量修改 | Claude Code | 按反模式清单重写59个插件，声称"已按清单规范化" | ~43分钟 |
| 2. 遗漏发现 | Tony | 让Claude Code分析自己的产出物，发现3个遗漏 | 几分钟 |
| 3. 首轮修复 | Claude Code | 修复4个同类插件，版本升至1.0.2 | ~1分钟 |
| 4. 二次示范 | Tony | 手动修改load_bbs_templates，示范3个Claude Code未做到的细节 | — |
| 5. 经验沉淀 | Claude Code | 将互动经验写入技能文档 | ~1.5分钟 |

### 关键转折

Tony没有直接说"你漏了什么"，而是说：

> **"分析这个规范文件，是否存在问题"**

这迫使Claude Code**自己审查自己的产出**，而不是被动接受批评。

## 二、发现的遗漏模式

| 遗漏类型 | 具体表现 | 根因 |
|----------|----------|------|
| 目录/文件混淆 | 创建目录的插件未处理base_path是文件的情况 | 只关注了TOCTOU，忽略了反模式清单第3条 |
| outputs不完整 | definition.yaml未声明error_code/error_message | 只修改了execute.py，未同步检查definition.yaml |
| 错误码缺失 | execute.py返回了新错误但ErrorCode类和definition.yaml未声明 | 修改代码时只关注逻辑，未同步更新声明 |

## 三、Tony示范的3个细节（Claude Code没做到的）

1. **_ok()返回也包含error_code=None, error_message=None** — 统一返回结构
2. **_fail()日志带上[error_code]** — 便于grep搜索定位
3. **路径存在但非目录时返回E_PATH_INVALID** — 更精确的错误分类

## 四、管理者需要学习的5条能力

### 能力1：不要直接指出错误，让开发者自己审

**Tony的做法**：不说"你漏了目录/文件混淆"，而是说"分析这个文件是否存在问题"。

**原理**：
- 直接指出 → 开发者被动修复，不形成深层认知
- 让开发者自审 → 主动发现遗漏，形成自查习惯
- 开发者自己发现的错误，记忆更深

**管理者口诀**：**"问问题，不给答案"**

### 能力2：批量产出后，必须触发逐条复查

**问题**：Claude Code批量修改59个插件后，声称"已按清单规范化"——但实际有遗漏。

**根因**：批量修改时容易"机械应用"清单条目，每条在不同上下文中的具体落实方式不同。

**管理者动作**：
1. 批量修改完成后，**随机抽检**1-2个产出物
2. 让开发者**逐条对照清单**检查每个文件
3. 发现遗漏后，**立即要求同类文件全部复查**

**口诀**：**"批量产出必抽检，抽检出问题必全检"**

### 能力3：耦合文件必须同步检查

**问题**：修改execute.py时，未同步检查definition.yaml。

**规律**：definition.yaml和execute.py是**一对耦合文件**：
- 改了execute.py的错误码 → definition.yaml的error_codes必须同步
- 改了execute.py的返回字段 → definition.yaml的outputs必须同步
- 改了definition.yaml的behavior → execute.py的实现必须对应

**管理者动作**：在规范中明确声明耦合关系，修改任一方时触发另一方检查。

**口诀**：**"改一查二，耦合必同步"**

### 能力4：示范比指令更有效

**Tony的做法**：不是告诉Claude Code"你的_ok()应该包含error字段"，而是**直接修改load_bbs_templates做示范**。

**示范的3个细节**都是Claude Code批量修改时没做到的：
- 统一返回结构
- 日志带错误码
- 更精确的错误分类

**原理**：
- 指令说"要统一返回结构" → 开发者理解但不知道具体怎么写
- 示范直接展示代码 → 开发者看到具体写法，直接复制模式

**口诀**：**"说一遍不如做一遍，示范一个胜过指令十条"**

### 能力5：经验必须立即沉淀，否则下次还会犯

**Tony的要求**：**"能把我们刚才的互动经验和流程总结一下吗？可能变成一种SKILL吗？"**

**Claude Code的沉淀**：
- `general_patterns.md` — 新增"批量代码修改后的自审流程"（4步流程+遗漏模式表）
- `project_specific.md` — 新增3条踩坑记录

**原理**：
- 不沉淀 → 下次批量修改还会犯同样的遗漏
- 沉淀到技能文档 → 下次开发时自动加载，形成防护网
- 沉淀到anti-patterns.md → 所有开发者共享经验

**口诀**：**"踩坑必记录，记录必沉淀，沉淀必复用"**

## 五、闭环流程总结

```
开发者批量产出
    ↓
管理者抽检（不直接指出，让开发者自审）
    ↓
开发者发现遗漏 → 立即修复 + 同类全检
    ↓
管理者示范更高标准（做一遍，不只说一遍）
    ↓
开发者学习示范 → 修复剩余
    ↓
管理者要求沉淀经验 → 写入技能文档/反模式清单
    ↓
下次开发自动加载经验 → 质量螺旋上升
```

## 六、Atlas训练开发者的操作方法

### 核心思路

拿**修改前的代码 + 修改意见 + 修改后的代码**，就能训练开发者。不需要Tony亲自下场。

### 具体操作

1. **准备素材**：从pipeline/done（修改前）和pipeline/done_claude（修改后）提取同一插件的对比
2. **喂给Claude Code子代理**：
   - 给他看修改前的代码
   - 给他修改意见（不是代码，是原则性指导）
   - 让他自己重写
   - 对比修改后的代码，看差距
3. **抽检自审**：让他分析自己的产出是否存在问题
4. **示范提升**：展示更高标准的写法，让他学习
5. **沉淀经验**：要求写入技能文档

### 素材来源

| 素材 | 位置 | 用途 |
|------|------|------|
| 修改前代码 | `pipeline/done/<组>/<插件>/` | 原始版本 |
| 修改意见 | `pipeline/done/<插件>/review_claude_opus.txt` | Opus的反馈 |
| 修改后代码 | `pipeline/done_claude/<组>/<插件>/` | Claude Code重写版 |
| 反模式清单 | `C:\tower-of-babel\projects\civibbs\dev-docs\standards\anti-patterns.md` | 14条规范 |
| Tony示范 | `pipeline/done_claude/bb/` (v1.0.2~1.0.4) | 更高标准的写法 |

### 训练流程

```
1. 选一个未评审的插件（如pt/generate_id）
2. 给Claude Code看：
   - done/pt/generate_id（修改前，1.0.0）
   - anti-patterns.md（14条规范）
   - 不给done_claude版本（让他自己写）
3. 让他重写
4. 对比done_claude版本，找差距
5. 让他自审："分析你的产出，是否存在问题"
6. 展示done_claude版本作为示范
7. 要求沉淀经验
```
