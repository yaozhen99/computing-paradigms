# 网页聊天自动化：从降维提取到拟人执行

> 作者：Tower of Babel 团队 | 日期：2026-05-04
> 状态：V1 已验证，DeepSeek 实测通过

## 摘要

本文提出并验证了一套 AI 智能体操作网页聊天界面的完整方案。核心思想是**将"理解页面"与"操作页面"彻底分离**：用降维提取技术将动态网页冻结为静态文本再离线解析，用行为模拟技术在实时环境中像真人一样读写。该方案在 DeepSeek 网页端完成了端到端验证，无需任何平台特定适配即可通用。

---

## 1 问题

AI 智能体（如 Hermes Agent）需要与网页聊天界面交互——在 DeepSeek、ChatGPT、智谱等平台上读取消息、输入回复、持续对话。这面临三重困难：

1. **DOM 脆弱性**：现代前端使用混淆类名（如 `d29f3d7d`）、动态渲染、字符级切碎，任何基于 CSS 选择器的提取方案都会在前端发版后失效。
2. **反自动化检测**：网站通过行为分析（打字速度、鼠标轨迹、请求频率）识别机器人，直接注入 `value` 或批量操作会触发风控。
3. **平台多样性**：每个网站的 DOM 结构、API 接口、交互逻辑都不同，逐个适配维护成本极高。

传统方案各有盲区：

| 方案 | 问题 |
|------|------|
| DOM 选择器 + 平台适配 | 类名混淆失效，维护成本高 |
| 纯视觉 AI（截图+OCR） | 响应慢（秒级），成本高 |
| API 逆向 | 接口随时变，法律风险 |
| JS 注入（Hook/拦截） | 触发反调试，崩溃风险 |

---

## 2 核心思想：侦察与行动分离

本方案的基石是一个架构层面的分离：

```
┌─────────────────┐     配置/知识     ┌─────────────────┐
│   离线侦察阶段    │ ──────────────→  │   实时执行阶段    │
│  (本地，零风险)   │                   │  (在线，拟人化)  │
│                  │                   │                  │
│  保存页面快照     │                   │  读取：降维提取   │
│  暴力分析结构     │                   │  写入：行为模拟   │
│  提取稳定标识     │                   │  等待：内容稳定   │
└─────────────────┘                   └─────────────────┘
```

**离线阶段**在本地对页面快照做任意"有损"分析——反复调试、暴力正则、AI 语义理解——因为不触碰真实网站，零风控风险。

**实时阶段**只执行最精简、最像人的操作：读文本、打字、按回车。不注入代码，不修改 DOM，不触发反调试。

## 3 降维提取：读的策略

### 3.1 物理断点

Tony 提出的"降维提取"核心是制造**物理断点**：将动态的、充满陷阱的网页，瞬间转化为静态的、死寂的本地文本。

具体实现：Playwright 的 `page.inner_text("body")` 一次性提取页面全部可见文本。这一步：

- **抹杀 DOM 结构**：所有 div/span/svg 标签消失
- **抹杀混淆类名**：`d29f3d7d`、`_63c77b1` 等不再存在
- **抹杀反调试陷阱**：debugger 循环、Context 隐藏全部失效
- **保留语义信息**：用户消息、AI 回复、思考过程完整保留

### 3.2 语义解析

从纯文本流中提取对话，不依赖任何 DOM 特征，而是依赖**跨平台通用的语义标记**：

| 标记 | 含义 | 适用平台 |
|------|------|----------|
| "已思考（用时 X 秒）" | AI 回复开始 | DeepSeek |
| "Thinking..." | AI 回复开始 | ChatGPT |
| "思考过程" | AI 回复开始 | 智谱 |
| 短行（< 80 字符） | 用户消息 | 通用 |

解析规则极简：遇到 AI 标记→切换为 assistant 角色，否则→user 角色。即使标记变化，只需改一行正则，不需要重写整个适配层。

### 3.3 快照存档

每次读取自动将页面文本存入 `logs/` 目录，形成离线档案。这有两个好处：

1. **审计追溯**：可回溯任何一次对话的完整页面状态
2. **离线调试**：在本地对存档文件反复调试解析逻辑，不影响线上

### 3.4 实测验证

在 DeepSeek 上实测，`page.inner_text("body")` 返回 2031 字符的纯文本，包含完整对话：

```
那5000-69呢
已思考（用时 1 秒）
我们需要计算5000 - 69。5000 - 69 = 4931。
求函数 f(x) = x^3 - 3x^2 + 2 在区间 [0,3] 上的最大值和最小值
已思考（用时 3 秒）
我们要求函数 f(x) = x^3 - 3x^2 + 2 在区间 [0,3] 上的最大值和最小值。
首先求导数：f'(x) = 3x^2 - 6x = 3x(x-2)...
```

数学公式的 LaTeX 渲染被切碎为单字符（`𝑓`、`(`、`𝑥`、`)`），但这恰好印证了 Tony 的判断：**AI 不依赖缩进，不依赖语法高亮，只依赖语义**。切碎的公式对人类阅读不友好，但对 AI 语义理解无影响。

## 4 行为模拟：写的策略

### 4.1 为什么不能直接注入

直接设置 `textarea.value = "hello"` 对 React/Vue 框架无效——框架维护自己的虚拟 DOM 状态，不会感知到原生 DOM 变更。实测中，这种方式在 DeepSeek 上输入框显示空白，消息未发送。

Playwright 的 `fill()` 和 `type()` 方法会触发完整的浏览器事件链（keydown→keypress→input→keyup），框架能正确捕获。

### 4.2 拟人化参数

基于"高度拟人化"方案的要求，输入行为模拟真人打字：

| 参数 | 值 | 依据 |
|------|-----|------|
| 逐字延迟 | 40-120ms | 正常打字速度范围 |
| 标点停顿 | 200-500ms | 标点后自然停顿 |
| 思考停顿 | 3% 概率，0.5-1.5s | 模拟犹豫/回忆 |
| 发送前停顿 | 300-800ms | 审视消息 |

这些参数使每次发送的总耗时不同，避免被频率分析识别。

### 4.3 发送机制

按 Enter 键发送，同时尝试点击发送按钮（如果存在）。双保险策略确保在不同平台上都能成功发送。

## 5 回复完成检测：等的策略

### 5.1 不依赖 DOM 指示器

传统方案查找"停止生成"按钮来判断回复是否完成，但这依赖特定选择器，平台变化即失效。

### 5.2 内容稳定性检测

采用降维提取的同一思路：每隔 0.8 秒提取页面文本的 MD5 hash，当 hash 连续 2 秒不变时，判定回复完成。

这比 DOM 指示器更可靠：
- 不依赖任何 CSS 选择器
- 不依赖按钮是否存在
- 即使平台没有"停止生成"按钮也能工作
- 天然处理流式输出（内容还在变→hash 变化→继续等待）

### 5.3 快速路径

作为优化，仍会检查"停止生成"按钮是否可见。如果可见，直接跳过 hash 检测，节省等待时间。但这只是优化，不是必需路径。

## 6 系统架构

```
webchat.py (CLI 入口)
  ├── connect    → 检查 CDP 端口 9222
  ├── dump       → DOM 探测（辅助工具）
  ├── read       → page.inner_text() → 语义解析 → 返回消息列表
  │                  └→ 自动存档到 logs/
  ├── send       → 找输入框 → 拟人化 type() → Enter 发送
  ├── wait       → 循环 hash 检测 → 回复完成 → 提取最后回复
  └── chat       → send + wait 一条龙
```

**零平台适配**：没有 adapters/ 目录，没有平台特定 JSON。所有平台走同一条路径。

**依赖**：Playwright（连接已有浏览器实例），websockets（CDP 连接），无其他外部依赖。

## 7 方案对比

| 维度 | DOM 选择器方案 | 纯视觉 AI 方案 | 本方案（降维提取+行为模拟） |
|------|---------------|---------------|---------------------------|
| 读取方式 | querySelector | 截图+OCR | inner_text 降维提取 |
| 角色识别 | CSS 类名 | 视觉判断 | 语义标记（"已思考"等） |
| 输入方式 | fill() 或 value 注入 | 坐标点击 | type() 逐字模拟 |
| 平台适配 | 每个平台一个 JSON | 无需 | 无需（通用语义标记） |
| 维护成本 | 高（类名随时变） | 低 | 极低（改正则即可） |
| 响应速度 | 快 | 慢（秒级） | 快 |
| 反检测能力 | 弱 | 强 | 强 |
| 数学公式 | 完整保留 | OCR 可能出错 | 切碎但语义完整 |

## 8 局限与演进

### 8.1 当前局限

1. **语义标记依赖**：如果某平台 AI 回复不以"已思考"/"Thinking"开头，角色识别可能失效。但降维提取的文本流中，用户消息通常短、AI 回复通常长，即使没有标记也能用长度启发式区分。
2. **数学公式切碎**：LaTeX 渲染的公式被拆成单字符节点，对人类阅读不友好。但消费者是 AI，语义理解不受影响。
3. **侧边栏噪声**：`inner_text("body")` 包含侧边栏历史记录等无关内容。当前通过定位聊天区域（AI 标记附近）过滤，但不完美。

### 8.2 演进方向

1. **网络层拦截**（上策）：监听 WebSocket/API 响应，直接获取结构化消息数据，绕开页面渲染。这是"高度拟人化方案"中提出的最优读取策略。
2. **MutationObserver**（中策）：被动监听 DOM 变化，只在内容增加时触发通知，实现毫秒级实时感知。
3. **鼠标轨迹模拟**：集成 ghost-cursor 等库，让鼠标移动产生贝塞尔曲线轨迹，进一步增强拟人度。
4. **触发器引擎**：定时/事件/条件/链式触发，实现自动监控和主动对话（V2 规划）。

## 9 结论

本文提出的"降维提取+行为模拟"方案，通过将页面理解与页面操作分离，实现了：

- **通用性**：零平台适配，DeepSeek/ChatGPT/智谱走同一条代码路径
- **稳定性**：不依赖混淆类名，前端升级不影响提取逻辑
- **安全性**：不注入代码，不修改 DOM，不触发反调试
- **拟人度**：逐字输入+随机停顿+标点延迟，模拟真人打字

该方案已在 DeepSeek 网页端完成端到端验证，从简单算术到微积分证明均成功收发。核心思想源自 Tony 的"降维暴力提取"和"离线侦察+实时执行"框架，本论文将其落地为可运行的工程实现。
