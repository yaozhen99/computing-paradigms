# R8 审查：用三公理重审当前实现

> 版本：v1.0

> 作者：yaozhen99 + Claude Opus 4.7
> 状态：审查完成，方向修正待定

## 0. 审查结论

R8 的 ISA 管道用传统 AI 应用的方式（对话记忆 + 上下文注入）解决了 StateKanban 的问题。但 StateKanban 本身就是要替代传统方式的——用 Claude Code 的方式做 Claude Code 已经在做的事，没有意义。

**R8 的代码不是废的，但位置错了。**

---

## 1. 三公理审查

### 公理 1：LLM 降级为无状态算力单元

R8 做了什么：ConversationMemory 把 LLM 的对话历史存下来，下次 drive 时注入回去。

问题：这不是"无状态"，是**给 LLM 偷偷塞状态**。LLM 本该每次调用都是白纸一张，靠看板信号决定行为。现在 LLM 有了"记忆"，它开始依赖自己的历史，而不是只依赖看板上的信号。

方向相反。Final Spec 说 LLM 不负责存储状态、不维护上下文。R8 的 ConversationMemory 恰恰在帮 LLM 维护上下文。

### 公理 2：状态看板是唯一真理源

R8 做了什么：Briefing 从 ConversationMemory 提取交接信息，注入下一个 LLM 调用。

问题：现在有**两个真理源**——看板和 ConversationMemory。LLM 看到的上下文一部分来自看板信号，一部分来自 memory 注入。哪个是权威？冲突时听谁的？

正确做法：**所有状态都在看板上**。跨 drive 的上下文延续，应该是看板上的信号没被清空（或者 CrystalZone 固化了关键事实），而不是靠一个旁路的 memory 模块。

### 公理 3：指令级底座管理看板生命周期

R8 做了什么：Engine 每次调用 LLM 前后，memory 自动 append 和 inject。

问题：Engine 在管理 memory 的生命周期，但 memory 不是看板的一部分。它是一个**平行于看板的状态容器**，绕过了看板的生命周期管理。

正确做法：跨 drive 的状态延续，是**看板本身的生命周期问题**——看板不随 drive 销毁，drive 只是看板上的一次信号碰撞循环。看板在，状态就在。

---

## 2. 三个核心问题的解法对比

### 问题 1：处理流程无限延续

当前 Engine.drive() 是有边界的循环——开始、跑几轮、收敛、结束。drive 结束，进程就没了。

R8 ISA 方案：在进程外面绑一条记忆管（ConversationMemory）。进程本身还是生灭的，memory 只是让下一个进程"假装"自己是上一个的延续。

Final Spec 方案：**看板不灭，进程即生即灭**。流程的延续性由看板承载，不靠记忆管。看板上的信号和 artifact 一直在，新进程看一眼看板就知道之前发生了什么。

### 问题 2：单个进程任意取消

现在取消一个 drive，看板状态可能处于半成品——信号写了但没收敛。R8 的 memory 不会帮你恢复，因为 memory 记录的是对话历史，不是看板状态。

Final Spec 方案：**清空流态区，零成本无损接管**。杀掉进程，清空 FluidZone，看板回到上一次晶态固化的稳定点。新进程从稳定点重新开始。不需要 memory，不需要 briefing——看板本身就是恢复点。

### 问题 3：与底层解耦，可任意方向开发

R8 的 ConversationMemory 把 LLM 对话历史和 Python 对象绑死了——LLMMessage、LLMResponse 是 Python 进程内存里的。进程崩了，memory 就丢了（除非持久化到文件，但那就是在模拟一个数据库——而看板本来就是数据库）。

Final Spec 方案：**进程之间调用和通信不再依赖操作系统的要求，自成系统**。看板是通信层，信号是通信协议，角色是通信端点。不需要操作系统的进程间通信（IPC、文件、网络），因为所有东西都在看板上。看板就是操作系统。

## 3. 对比总表

| 问题 | R8 ISA 方案 | Final Spec 方案 |
|:---|:---|:---|
| 跨 drive 失忆 | ConversationMemory 记对话历史 | 看板不灭，信号和 artifact 持久 |
| 进程取消恢复 | 无解（memory 不记录看板状态） | 清空 FluidZone，从晶态稳定点重启 |
| 上下文延续 | Briefing 注入 system 消息 | 视口切片从看板读取当前状态 |
| 多进程协作 | 不支持 | 信号驱动，看板即通信总线 |
| 真理源 | 双源（看板 + memory） | 单源（看板唯一） |
| LLM 状态性 | 有状态（记忆注入） | 无状态（每次白纸，只看看板） |

## 4. R8 代码的可复用部分

R8 的代码不是废的。工程能力在，只是位置错了。

### 可复用

| 组件 | 当前位置 | 迁移方向 |
|:---|:---|:---|
| token 预算管理（_estimate_tokens, max_tokens） | ConversationMemory | ViewportSlicer 的 token budget 硬限制 |
| 自动摘要（_compact_if_needed, _summarize_messages） | ConversationMemory | 看板信号压缩——FluidZone 信号过多时自动摘要为 CrystalZone artifact |
| 结构化提取（_extract_role/task/progress/decisions） | Briefing | 看板元数据提取——从信号历史提取项目状态摘要 |
| 持久化（save/load JSON） | ConversationMemory | 看板快照——看板状态持久化的参考实现 |
| inject() 消息组装 | Briefing | 视口切片组装——将看板状态组装为 LLM 输入 |

### 不可复用（方向错误）

| 组件 | 原因 |
|:---|:---|
| ConversationMemory 作为 LLM 旁路记忆 | 违反公理 1（LLM 无状态）和公理 2（看板唯一真理源） |
| Briefing 注入 system 消息 | 应该通过视口切片从看板读取，不应旁路注入 |
| LLMMessage/LLMResponse 数据类 | 与 Python 进程内存绑死，应该用看板信号承载 |
| --memory on/off/auto CLI 参数 | 不应该有"关闭记忆"的选项——看板永远在，不需要开关 |

## 5. 方向修正：R8 应该做什么

R8 的核心任务不是"给 LLM 加记忆"，而是**让看板成为跨 drive 的状态承载者**。

### 修正后的 R8 目标

1. **看板生命周期与 drive 解耦** — drive 是看板上的一次信号碰撞循环，drive 结束看板不销毁
2. **FluidZone 清空恢复** — 进程取消时清空流态区，从晶态稳定点重启
3. **视口切片从看板读取完整状态** — 不需要 memory 注入，视口切片本身就能看到看板上所有信号和 artifact
4. **信号压缩** — FluidZone 信号过多时，自动摘要为 CrystalZone artifact（复用 R8 的摘要逻辑）

### 与 R9 的关系

R8 修正和 R9 信号驱动有重叠但不冲突：

- R8 修正：解决**看板作为唯一真理源**的问题（状态承载、生命周期、恢复）
- R9：解决**看板作为事件总线**的问题（信号驱动、即时通知、进程即生即灭）

R8 修正是 R9 的前提——如果看板不是唯一真理源，信号驱动就没有意义（信号驱动的前提是所有状态都在看板上）。
