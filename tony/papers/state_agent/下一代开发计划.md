## 0. 从哪里来

### 原始构想

《基于状态看板的指令级开发架构（Final Spec）》定义了三条核心公理：

1. **LLM 降级为无状态算力单元** — 不负责存储状态，不维护上下文，只做单点推理
2. **状态看板是唯一真理源** — 所有状态承载者，严格分为流态区（高频信号）和晶态区（稳定事实）
3. **指令级底座管理看板生命周期** — 进程实例化/销毁、视口切片、输出阀门

核心运行机制：**信号写入看板 → 碰撞检测 → 角色即时响应 → 收敛或新一轮**。这是事件驱动模型。

### 当前实现（R7 完成后）

R1~R7 完成了 StateKanban 内核、引擎、隔离、三路适配器的全量实现（596 测试通过）。但实现偏离了构想的核心运行机制：

| 构想 | 实现 | 偏离程度 |
|:---|:---|:---|
| 信号写入 → 即时触发角色响应 | Engine 按固定顺序遍历角色，每轮批量处理 | **核心偏离** |
| 进程即生即灭：实例化→调用→销毁 | 角色只是循环变量，没有实例化/销毁闭环 | 中度偏离 |
| 视口切片死死锁死在极小固定阈值 | ViewportSlicer 存在但 `_build_context()` 没执行截断 | 中度偏离 |
| 清空流态区 → 零成本无损接管 | `clear_signals()` 存在但未被调用，熔断只返回失败 | 中度偏离 |
| MessageBus 是实时通信层 | MessageBus 代码完整但无人使用，是空壳 | **核心偏离** |

**结论：当前系统建了一套事件驱动的架构，但用轮询的方式运行。看板是数据库，不是事件总线。**

## 1. 核心问题诊断

### 问题 1：看板是数据库，不是通信总线

Engine.drive() 的循环：

```python
while True:
    for role in [coder, reviewer, tester, integrator]:  # 顺序遍历
        slice = ViewportSlicer.slice(role)               # 从 kanban 读
        response = LLM.complete(slice)                   # 调 LLM
        parsed = ResponseParser.parse(response)           # 解析
        kanban.fluid.write_signal(parsed)                 # 写回 kanban
```

信号写入 FluidZone 后，没有任何机制通知相关角色。Reviewer 要等到下一轮才能看到 Coder 的信号。**看板的实时性被 Engine 的顺序循环扼杀了。**

### 问题 2：MessageBus 已建好但没人用

MessageBus 有 `publish()`/`subscribe()`/`sync_call()`/`async_notify()`——完整的通信基础设施。但 `cmd_drive()` 里没有注册任何订阅者，没有注册任何 handler。Engine 完全绕过了 MessageBus。

FluidZone 和 MessageBus 的关系应该是：
- FluidZone = **持久层**（信号写入后不丢失，可回溯）
- MessageBus = **实时层**（信号写入后立即通知订阅者）

当前只有持久层在工作，实时层是空壳。

### 问题 3：进程不是进程

ProcessManager 有完整的状态机（CREATED→ACTIVE→SUSPENDED→TERMINATED）、心跳、交接。但 `cmd_drive()` 只创建了一个 coder process。其他角色没有进程实例。

Final Spec 说"杀死进程仅仅是 del process_object"。当前实现里角色不是独立对象，是循环变量。

## 2. 解决方案：接通已有基础设施

核心变更不是重写 Engine，而是**接通已经建好但没用的基础设施**。

### 变更 1：FluidZone.write_signal() 触发 MessageBus.publish()

```python
class FluidZone:
    def write_signal(self, signal: Signal) -> None:
        self._validate_signal(signal)
        # ... 原有逻辑 ...
        self._signals.append(signal)
        # 新增：通知 MessageBus
        if self._bus is not None:
            await self._bus.publish(signal)
```

信号写入看板后，自动触发实时通知。角色不再等"下一轮"，而是被信号驱动。

### 变更 2：角色注册订阅

```python
# cmd_drive() 里：
bus.subscribe("intent", coder_callback)
bus.subscribe("intent", reviewer_callback)
bus.subscribe("veto", coder_callback)
bus.subscribe("error", coder_callback)
```

每个角色订阅自己关心的信号类型。收到信号后自主决定是否响应。

### 变更 3：Engine 从调度者变成看门人

Engine 不再决定角色执行顺序，只负责：
- 收敛检测（信号碰撞是否解决）
- 熔断保护（超过最大轮次）
- 晶态固化 + 阀门输出

角色的执行顺序由信号驱动自然形成：Coder 写 IntentSignal → Reviewer 收到通知 → 自主响应。

### 变更 4：进程即生即灭

`_process_role()` 改成：
```python
process = Process(role=role)          # 实例化
slice = slicer.slice(role)            # 视口裁剪
response = await call_llm(slice)      # 无状态调用
kanban.fluid.write_signal(parsed)     # 信号回传
del process                           # 销毁
```

每次 LLM 调用前后，进程完整地生灭一次。没有跨调用的状态残留。

### 变更量估计

200-300 行修改，不需要新增模块。MessageBus、ProcessManager、ViewportSlicer、ConvergenceDetector 都已经存在。

## 3. 多看板智能体：跨项目协作

### 核心洞察

跨项目协作的驱动力是**接口适配**——项目 A 暴露了接口，项目 B 依赖这个接口，接口偏移了就要协调。

### 三种跨项目通信模式分析

| 模式 | 描述 | 问题 |
|:---|:---|:---|
| 直通 | A 直接发信号给 B | B 在忙怎么办？没有仲裁 |
| 大看板 | 所有项目信号汇聚到上层 Meta-Kanban | 信号语义冲突，项目内和项目间的信号类型打架 |
| **协调看板** | 专门起一个看板智能体 C 管 A↔B | **无问题——C 就是另一个看板，恰巧 A 和 B 也是看板** |

### 选定方案：协调看板智能体

C 不是 A 和 B 的上级，C 是 A 和 B 之间的**同行**。C 自己也是一个完整的看板——有自己的 FluidZone、角色、信号。C 的任务恰好是"协调 A 和 B 的接口适配"。

C 操作 A 和 B 的方式，和 coder 操作代码文件的方式一样——通过视口切片看到目标，通过信号表达意图，通过阀门执行变更。只不过 C 的"代码文件"是 A 和 B 的 CrystalZone 里的接口契约。

**关键设计决策：整个体系只有一套机制，没有例外。**

项目内协作、项目间协调、元协调，全是同一个循环：

```
看板 → 视口切片 → LLM → 信号 → 碰撞 → 收敛
```

### 看板间通信的技术实现

所有看板在同一个 Python 进程内存里。看板间通信不需要走网络、走 IPC、走文件——就是函数调用级别（纳秒延迟）。

需要新增一个 `KanbanRouter`：

```python
class KanbanRouter:
    """看板间信号路由器——同一进程内的内存路由"""
    
    def register(self, kanban: StateKanban):
        self._kanbans[kanban.name] = kanban
    
    def subscribe(self, source: str, signal_type: type, handler: Callable):
        """看板 B 订阅看板 A 的某类信号"""
        ...
    
    def route(self, signal: Signal, source: str):
        """信号从 source 看板发出，路由到所有订阅者"""
        ...
```

两层架构：

```
KanbanRouter（看板间）
  ├── 看板 A
  │     └── MessageBus（角色间：coder ↔ reviewer ↔ tester）
  ├── 看板 B
  │     └── MessageBus（角色间：coder ↔ reviewer ↔ tester）
  └── 看板 C（协调 A↔B）
        └── MessageBus（角色间：contractWatcher ↔ adaptationCoordinator）
```

### 协调看板的生命周期

协调看板是**即生即灭的**。A 和 B 的接口稳定了，C 就销毁。不需要常驻的 Meta-Kanban。

新项目 D 也依赖 A 的接口？再起一个协调看板管 A↔D。互不干扰。**随时开新信道，因为都在虚拟层上面跑，底层管不着。**

## 4. 实现路线图

### R8（当前进行中）：记忆层 + Briefing 交接

ISA 流程已在执行。内容：
- ConversationMemory（对话记忆层，token 预算 + 自动摘要 + 持久化）
- Briefing（适配器交接上下文生成器）
- CLI `--memory` / `--briefing` 参数
- Engine 集成 memory 和 briefing 注入

**注意：R8 的 memory/briefing 是在轮询模式下实现的。后续架构修正（信号驱动）后，session 的语义会从"一次 drive 调用的执行记录"变为"一次信号碰撞到收敛的完整轨迹"。R8 的代码需要适配，但核心数据结构（ConversationMemory、Briefing）可以复用。**

### R9：信号驱动架构修正

**这是最关键的一轮。** 接通已有基础设施，让看板从数据库变成事件总线。

变更清单：
1. FluidZone.write_signal() → MessageBus.publish()
2. 角色注册订阅（cmd_drive 里加 subscribe 调用）
3. Engine 从调度者变成看门人（drive() 改成等待信号 → 检查收敛 → 熄断保护）
4. 进程即生即灭（_process_role 改成实例化→调用→销毁闭环）
5. 视口切片截断逻辑补全（_build_context 执行 token budget 硬限制）
6. FluidZone.clear_signals() 接入熄断恢复（卡死时清空重来）

验收标准：
- 信号写入后，订阅角色在**同一轮**内收到通知并响应（不是下一轮）
- 进程每次 LLM 调用前后完整生灭
- 视口切片 token 量硬限制在 max_tokens 以内
- 熄断时清空流态区可恢复
- R8 的 596+ 测试无回归

### R10：KanbanRouter + 协调看板原型

新增看板间信号路由层，验证协调看板智能体可行性。

变更清单：
1. 新增 `core/router.py` — KanbanRouter（看板间信号路由）
2. StateKanban 新增 `name` 属性和 `_bus_ref`（引用 KanbanRouter）
3. CLI 新增 `--kanban-name` 参数（命名看板实例）
4. 新增 `ContractSignal` / `AdaptationSignal` 跨项目信号类型
5. 协调看板原型：两个项目看板 + 一个协调看板，跑通一次接口适配

验收标准：
- 看板 A 写 ContractSignal → KanbanRouter 路由到看板 B → B 收到 AdaptationSignal
- 协调看板 C 能跑通一次完整的接口适配收敛
- 协调看板销毁后，A 和 B 的接口状态保留在各自的 CrystalZone

### R11：视口切片跨看板

当前 ViewportSlicer 只看单个看板的信号和 artifact。协调看板 C 需要同时看 A 和 B 的接口状态。

变更清单：
1. ViewportSlicer 支持多看板视口（指定 source_kanban 列表）
2. KanbanRouter 提供 `get_cross_kanban_slice()` 方法
3. 协调看板的角色视口包含 A 和 B 的 CrystalZone 接口契约

## 5. 架构自相似性

整个体系是自相似的——项目内协作和项目间协作用同一套机制：

| 层级 | 看板 | 角色 | 信号 | 收敛条件 |
|:---|:---|:---|:---|:---|
| 项目内 | 项目看板 A | coder/reviewer/tester | IntentSignal/VetoSignal/ErrorSignal | 代码通过审查 |
| 项目间 | 协调看板 C | contractWatcher/adaptationCoordinator | ContractSignal/AdaptationSignal | 接口对齐 |
| 元协调 | 元协调看板 D | scheduleCoordinator | ScheduleSignal | 多项目排程一致 |

每一层都是：**看板 → 视口 → LLM → 信号 → 碰撞 → 收敛**。没有例外，没有特殊机制。

Harness 文章的五层（边界层、记忆层、交接层、认知层、技能层）对应 StateKanban 的：

| Harness 层 | StateKanban 对应 | 当前状态 |
|:---|:---|:---|
| 边界层 | OutputValve + pipe_guard | 已实现（R6） |
| 记忆层 | ConversationMemory + Briefing | R8 开发中 |
| 交接层 | Briefing + session 日志 | R8 开发中 |
| 认知层 | **不需要** — 多角色架构天然解决"卡住换思路" | 构想层面已决定不采纳 |
| 技能层 | ISA 流程 + prompt 模板 | 已实现 |

**不采纳的部分**（与昨天分析一致）：
- 7 种推理模式 — 多角色架构是组织级解决方案，比个体级推理模式切换更可靠
- 双轨并行 — 当前只走轨道 A（在现有架构上迭代），轨道 B（先天长鞍）通过信号驱动架构修正实现
- 四工具记忆分工（grep/向量/图谱/git log） — 当前只做轻量方案，不上 RAG

## 7. 风险

| 风险 | 影响 | 缓解 |
|:---|:---|:---|
| R9 信号驱动改动导致 R8 记忆层返工 | 中 — session 语义变化 | R8 的 ConversationMemory/Briefing 数据结构可复用，只需适配调用时机 |
| 协调看板需要 LLM 判断接口偏移影响范围 | 中 — 增加一次 LLM 调用开销 | 协调看板可以用更便宜的模型（DeepSeek flash），不需要 Opus |
| 多看板在同一进程内存，看板间信号路由可能引入循环依赖 | 低 — A→C→B→A | KanbanRouter 加循环检测：信号最多路由 3 次 |
| 视口切片跨看板时 token budget 管理复杂 | 中 — 两个看板的信号量叠加 | 协调看板的视口只看接口契约（极小），不看完整代码 |
