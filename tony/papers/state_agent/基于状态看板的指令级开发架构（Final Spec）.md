基于状态看板的指令级开发架构（Final Spec）
核心公理：状态与计算彻底剥离
本系统的第一性原理是：大模型（LLM）不负责存储任何状态，不负责维护上下文，不作为系统的主体。
大模型被降级为“无状态的算力单元（ALU）”。
系统的唯一真理源、所有状态的承载者，是内存中的一个数据结构：状态看板。

一、 系统三大组件
1. 状态看板
整个系统唯一的数据中心，持久存在于内存中。严格分为两个区域：

流态区：存放高频、短生命周期的“意图信号”和“否决信号”。（例如：[Coder Intent 计划在 X 文件增加变量]，[Reviewer Reject 违反安全规则]）。
晶态区：存放低频、高稳定性的“事实代码块”。只有经过流态区充分碰撞、未被否决的代码草案，才能固化到晶态区。
2. 指令级底座
系统的“操作系统内核”，由纯确定性代码（PythonGoRust）编写。它的职责不是写代码，而是管理看板的生命周期：

进程实例化与销毁：内部逻辑进程（如 Coder, Reviewer）只是内存中的轻量对象。杀死进程仅仅是 del process_object，没有任何上下文需要交接或保存。
视口切片：在每次调用外部算力前，底座负责从看板上“裁剪”出一个极小的局部上下文。
输出阀门：物理世界的唯一出口。只有晶态区的代码，才允许被阀门写入真实的本地硬盘。
3. 外部大模型 API
纯粹的工具。接收底座传入的“视口切片”，执行一次单点逻辑推理，返回一条极短的“意图标签”或“代码片段”，随即断开连接。它不知道过去发生了什么，也不知道未来要做什么。

二、 核心运行机制：一次迭代的绝对闭环
当接收到一个开发需求时，系统的运行流如下：

需求注入：底座将需求解析为初始信号，写入看板的流态区。
实例化进程 A：A = Process(role=Coder)。
视口提取：底座从看板裁剪出当前切片（例如：需求信号 + 晶态区当前文件指针）。注意：切片极小，绝不包含冗长的历史对话。
无状态调用：将切片发送给外部 LLM API。
信号回传：LLM 返回一个意图（如：“建议添加函数 Foo”）。底座将此意图写入看板流态区。
销毁进程 A：del A。进程 A 死亡，释放资源。
实例化进程 B：B = Process(role=Reviewer)。
视口提取：底座重新裁剪切片（包含刚才 Coder 的意图 + 安全规则约束）。
无状态调用：发给 LLM API。
信号碰撞：如果 LLM 返回否决信号，写入流态区，触发新一轮循环。如果返回通过，该代码草案从流态区降级进入晶态区。
物理输出：底座阀门检测到晶态区有新代码，调用本地写入工具，更新真实文件。
三、 为什么这个架构能解决长程开发的绝症？
彻底解决上下文溢出：因为不存在长上下文。外部 API 每次处理的 Token 数量被底座的“视口切片”死死锁死在一个极小的固定阈值内。
彻底解决错误累积（死循环）：传统 Agent 带着错误记忆越走越偏。本系统中，错误只存在于看板的流态区某一条信号里。一旦被否决信号覆盖，下一次 API 调用的视口里就再也没有那个错误了，实现了绝对的环境自愈。
零成本的无损接管：当一个任务卡死，底座可以直接清空看板流态区，重新实例化一组进程。没有任何历史包袱需要清理，实现了真正的“进程平滑生死”。
四、 工程测试标准（摆脱黑盒测试）
不要用“最后生成的代码能不能跑”来测试这个系统。要将系统视为一个力学结构，测试其拓扑属性：

收敛率测试：Mock 掉外部 LLM（让它返回随机固定格式），单纯测试内部看板规则。一个需求注入后，流态区经过多少次信号碰撞能稳定输出到晶态区？震荡次数必须可控。
拦截率测试：故意让 Mock LLM 返回包含恶意代码的意图，测试看板规则或审查进程能否 100% 拦截，确保晶态区零污染。
状态无损交接测试：在进程 A 写入看板一半时强制 del A，唤醒进程 B，验证 B 提取到的视口切片是否精准包含了 A 遗留的关键信号，没有信息断崖。
