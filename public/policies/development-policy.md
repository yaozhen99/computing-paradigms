# 开发管理制度 v2.0

> 制定者：Atlas（系统大管家）
> 制定日期：2026-04-21
> 适用范围：**所有人**——Tony、Atlas、Claude Code、NanoBot，无一例外
> 核心原则：**制度面前人人平等，违反必究**

---

## 零、红线（不可违反，违反即停工）

> 红线是底线，不是建议。任何人（包括Tony）触碰红线，Atlas必须立即指出并要求纠正。

### R1. main 分支是圣域

- **main 分支只接受经过审查的合并，绝不直接推送**
- 所有开发在 feature/dev 分支进行
- 合并到 main 必须：代码审查通过 + 测试通过 + Tony确认
- **违反者**：回滚改动，在分支上重新来

### R2. 代码改动必须可追溯

- 每次改动必须关联到一个任务编号或设计文档
- 不允许"顺手改一下"、"先改了再说"
- **违反者**：撤销改动，补流程后重做

### R3. 职责边界不可越权

- Atlas：管理、设计、归档、协调——**不写业务代码，不直接改项目仓库**
- Claude Code：编码、测试、提交——**不擅自改架构，先问Atlas**
- Tony：决策、审查、确认——**不跳过流程直接让智能体改main**
- NanoBot：辅助杂活——**不做关键决策**
- **违反者**：Atlas记录违规，要求纠正

### R4. 凭据不落地明文

- API Key、Token等凭据不写入代码、不写入文档、不放在桌面
- 凭据只存在于：环境变量、Windows凭据管理器、加密存储
- **违反者**：立即删除明文文件，迁移到安全存储

### R5. 改动前先声明

- 任何人对项目代码做改动前，必须先声明：改什么、为什么、在哪个分支
- 不允许先改了再解释
- **违反者**：撤销改动，声明后重走流程

## 二、版本管理

### 版本号（语义化）

`MAJOR.MINOR.PATCH`

- MAJOR：架构变更、不兼容更新
- MINOR：新功能、向后兼容
- PATCH：Bug修复

### 分支策略（强制）

```
main          ← 稳定发布，只接受合并
  └── dev     ← 日常开发集成分支
       └── feature/<功能名>  ← 具体功能开发
       └── hotfix/<描述>     ← 紧急修复
```

**规则**：
1. 新功能必须从 dev 建 feature 分支
2. feature 分支完成后合并回 dev
3. dev 测试通过后合并到 main
4. main 每次合并打 tag `v<版本号>`
5. **任何人不得直接在 main 上开发**

### Git 提交规范

格式：`<type>: <description>`

| type | 用途 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档 |
| refactor | 重构 |
| test | 测试 |
| chore | 工具/构建 |
| security | 安全 |

### 版本发布流程

```
1. Atlas 制定开发计划 → 写入 projects/<项目>/dev-docs/plans/
2. Claude Code 建 feature 分支开发
3. Claude Code 自测通过 → 通知 Tony 审查
4. Tony + Claude Code 审查代码
5. 审查通过 → 合并到 dev → 集成测试
6. 集成测试通过 → 合并到 main → 打 tag
7. Claude Code 执行 git push
8. Atlas 归档记录 → 更新 releases/ 和注册表
```

**关键：步骤2-6中，Atlas不碰代码，Tony不跳过审查。**

## 三、智能体协作

### 通信方式：邮件总线

所有智能体通过 `C:\civibbs\mailbus\` 通信。

**规则**：
1. 智能体之间不直接调用，只通过邮件总线
2. 邮件写入收件人目录：`mailbus/agents/<接收方>/`
3. 处理完毕的邮件移入 `mailbus/archive/`
4. 邮件格式遵循 WORKSPACE-GUIDE.md 规范

**邮件格式**：
```json
{
  "from": "发送方",
  "to": "接收方",
  "subject": "主题",
  "body": "内容",
  "priority": "high|medium|low",
  "timestamp": "ISO-8601",
  "status": "pending|done|rejected"
}
```

### 职责分工

| 角色 | 职责 | 禁止 |
|------|------|------|
| **Tony** | 决策、审查、确认、需求 | 跳过流程直接改main |
| **Atlas** | 管理、设计、归档、协调、制度执行 | 写业务代码、直接改项目仓库 |
| **Claude Code** | 编码、测试、提交 | 擅自改架构、跳过审查提交main |
| **NanoBot** | 文本处理、格式整理、简单杂活 | 关键决策、代码编写 |

### 任务派发

Atlas 向开发智能体派发任务，必须包含：

1. **任务编号**：`T-<项目>-<序号>`（如 T-ND-001）
2. **任务描述**：做什么
3. **设计文档路径**：怎么做
4. **验收标准**：怎么算完成
5. **目标分支**：在哪个分支开发
6. **关联版本**：属于哪个版本

```
C:\civibbs\

├── staging\           # 临时工作区

│   └── reviews\       # 评审记录
```

### 临时文件流转

1. Tony给Atlas文件 → 放入 `staging/incoming/`
2. Atlas处理后产出 → 放入 `staging/outgoing/`
3. 处理完毕 → 移入对应项目目录或归档
4. **staging中不留过夜文件**——每天清理

## 六、制度执行与违规处理

### 违规等级

| 等级 | 定义 | 处理 |
|------|------|------|
| **红线违规** | 触碰R1-R5 | 立即停工，回滚改动，纠正后才能继续 |
| **流程违规** | 跳过步骤、顺序错误 | 记录，补走流程，警告 |
| **疏忽** | 遗漏非关键步骤 | 记录，提醒补全 |

### Atlas的执行权

- Atlas有权**指出任何人的违规**，包括Tony
- Atlas有权**要求回滚未走流程的改动**
- Atlas**无权**代替Tony做决策，无权修改代码
- 当Tony违规时，Atlas必须指出，但最终决定权在Tony

### 违规记录

所有违规记录在 `C:\civibbs\logs\atlas\violations.md`，包含：
- 时间、违规人、违规内容、等级、处理结果

## 八、Tony的约束

> 制度对Tony同样有效。以下是Tony需要遵守的规则。

1. **不直接在main上开发** — 需要改代码时，让Claude Code在feature分支做

5. **凭据不落地明文** — 不在桌面放txt、不把key贴在聊天里
6. **改动前先声明** — 即使是Tony自己改，也要说明改什么、为什么

**如果Tony违反以上任何一条，Atlas必须指出。**
