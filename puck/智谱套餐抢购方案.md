# 智谱套餐定时抢购脚本 — 方案文档

## 需求

在智谱AI开放平台（open.bigmodel.cn）浏览器端，定时（如10:00:00）精准点击"立即购买"按钮抢购限量套餐。抢到后锁单，人手完成支付。

## 核心策略

**脚本只做一件事：在毫秒级精度的时间点，向浏览器渲染窗口发送一次鼠标点击。**

- 不碰登录（手动提前登好）
- 不碰支付（锁单后人手操作）
- 不提前充值（锁单后再充也来得及）

## 技术路线：hwnd 窗口消息

### 为什么用 hwnd 而不是 API 抓包？

| 对比 | API抓包 | hwnd窗口消息 |
|------|---------|-------------|
| 速度 | 最快（直接HTTP） | 快（微秒级消息） |
| 逆向难度 | 需抓包分析接口、token | 只需按钮坐标 |
| 反检测 | 需伪造UA/cookie | 无WebDriver特征 |
| 稳定性 | 接口变了就废 | 页面布局变了才废 |
| 门槛 | 高 | 低 |

### 为什么用 hwnd 而不是 Selenium/Playwright？

- 不注入浏览器进程，无 WebDriver 特征
- 不需要安装驱动
- 消息直达，跳过物理输入设备延迟

## 窗口层级结构

```
顶级窗口：Chrome_WidgetWin_1（Chrome/Edge框架）
 └─ 渲染子窗口：Chrome_RenderWidgetHostHWND（网页内容区域）
```

**关键**：点击消息必须发给渲染子窗口，不是顶级窗口。否则坐标会因为标题栏/工具栏偏移。

## 时间精度方案

### 问题：本地时钟不可靠

系统时间可能偏差1-2秒甚至更多。

### 解决：NTP同步 + 忙等

1. 启动时请求NTP服务器（ntp.aliyun.com），计算本地时间与标准时间的偏差
2. 用偏差修正目标时间
3. 提前300ms激活窗口（窗口置顶需要50-200ms）
4. 最后0.3秒用CPU忙等（`while now < target: pass`），精度可达±5ms

### 为什么不用 time.sleep()？

`time.sleep()` 有1-15ms调度误差，忙等循环精度更高。

## 点击后检测与补点

### 三种状态

| 状态 | 页面表现 | 处理 |
|------|---------|------|
| 抢到（锁单） | 跳转确认订单页 | ✅ 清场退出 |
| 售罄 | 按钮变灰/提示已抢完 | ❌ 停止 |
| 拥挤/未提交 | 页面无变化 | ⚠️ 补点 |

### 检测方式：窗口标题变化

- 抢购前：标题含"智谱AI开放平台 - 套餐详情"
- 成功后：标题含"确认订单"或"订单详情"
- 售罄：标题含"售罄"或"已抢完"

每120ms检查一次标题，判断状态。

### 补点策略

- 最多补点15次，间隔120ms
- 总时长不超过5秒
- 检测到成功→清场退出
- 检测到售罄→停止
- 超时→提示人工检查

## 成功后清场（关键！）

抢到后必须立即停止一切自动化操作，否则残留的键盘/鼠标消息会干扰手动支付。

清场步骤：
1. 跳出所有循环，不再发送任何消息
2. 强制抬起修饰键（Ctrl/Shift/Alt）的KEYUP
3. 用PeekMessage+PM_REMOVE清空窗口消息队列中的残余键盘/鼠标消息
4. 停止窗口置顶操作
5. 干净退出

## 人抢不过机器的原理

| 环节 | 人 | 机器 |
|------|-----|------|
| 反应延迟 | 120-200ms | <5ms |
| 点击物理行程 | 50-100ms | 0（消息直达） |
| 点击频率 | 10-15次/秒 | 稳定8次/秒不衰减 |
| 认知循环 | 串行、受注意力影响 | 并行、无认知负担 |
| 窗口切换 | 50-200ms | 提前300ms完成 |

**本质**：人用生理极限对抗电子信号发生器，每个环节的延迟积分就是差距。

## 使用前清单

2. [ ] 确认浏览器标签页标题关键字
3. [ ] 确认抢购时间

## 风险提醒

- 代抢脚本可能违反平台用户协议，有封号风险
- 控制点击频率（≥100ms间隔），避免触发风控
- 仅供学习Windows API编程用途
